## 1. Зачем вообще появились СУБД

База данных нужна для того, чтобы хранить, изменять и извлекать данные не вручную из файлов, а через управляемую систему. СУБД берет на себя несколько важных задач:

- хранение данных на диске;
    
- быстрый поиск нужных записей;
    
- одновременную работу многих пользователей;
    
- контроль целостности данных;
    
- восстановление после сбоев;
    
- управление правами доступа.
    

Изначально данные часто хранились в обычных файлах, но по мере роста объемов и числа пользователей такой подход стал неудобным. Понадобилась система, которая отделяет логику работы приложения от физического хранения данных.

---

## 2. История и возникновение реляционных СУБД

### Революция Эдгара Кодда

В 1970 году Эдгар Кодд предложил реляционную модель данных. Главная идея состояла в том, что приложение не должно зависеть от того, как именно данные лежат на диске.

Это дало два важных эффекта:

- можно менять внутреннюю реализацию хранения без переписывания приложений;
    
- запросы к данным стали описывать не алгоритм, а требуемый результат.
    

### Основная идея реляционной модели

Реляционная модель опирается на строгую математическую основу — реляционную алгебру.

Базовые понятия:

- **отношение** — таблица;
    
- **кортеж** — строка;
    
- **атрибут** — столбец;
    
- **домен** — допустимое множество значений атрибута.
    

Пример: таблица `users(id, name, email)` — это отношение, каждая строка в ней — кортеж.

## 3. SQL и декларативный подход

SQL — это декларативный язык. Пользователь пишет, **что** он хочет получить, а не **как** это вычислить.

Пример:

```sql
SELECT name
FROM users
WHERE city = 'Moscow';
```

Здесь не указывается, идти ли по индексу, сканировать ли всю таблицу, сортировать ли промежуточные данные. Это решает оптимизатор СУБД.

### Почему это важно

Такой подход дает:

- независимость логики приложения от физического хранения;
    
- возможность оптимизации запросов самой СУБД;
    
- переносимость SQL-запросов между системами с похожим синтаксисом.
    

---

## 4. Основные операции реляционных СУБД

Реляционные базы особенно сильны там, где нужны сложные операции над связанными данными.

### 4.1. Фильтрация

Выбор строк по условию.

```sql
SELECT *
FROM orders
WHERE status = 'paid';
```

### 4.2. Проекция

Выбор только нужных столбцов.

```sql
SELECT id, total
FROM orders;
```

### 4.3. Join

Соединение данных из нескольких таблиц.

```sql
SELECT u.name, o.total
FROM users u
JOIN orders o ON u.id = o.user_id;
```

### 4.4. Агрегация

Подсчет итогов.

```sql
SELECT user_id, COUNT(*), SUM(total)
FROM orders
GROUP BY user_id;
```

### 4.5. Сортировка

Упорядочивание результата.

```sql
SELECT *
FROM orders
ORDER BY created_at DESC;
```

## 5. Проектирование данных и нормализация

Перед созданием таблиц обычно строят модель предметной области.

### ER-диаграмма

ER-диаграмма показывает:

- сущности;
    
- их атрибуты;
    
- связи между сущностями.
    

Например:

- `Пользователь`
    
- `Заказ`
    
- `Товар`
    

Связи:

- пользователь делает заказы;
    
- заказ содержит товары.
    

### Зачем нужна нормализация

Нормализация нужна, чтобы:

- уменьшить дублирование данных;
    
- избежать аномалий вставки, обновления и удаления;
    
- повысить логическую целостность.
    

### 1НФ

Все значения атомарны. В одной ячейке не должно быть списка значений.

Плохо:

|id|emails|
|---|---|
|1|[a@mail.com](mailto:a@mail.com), [b@mail.com](mailto:b@mail.com)|

Хорошо: отдельная таблица email-адресов.

### 2НФ

Нет частичных зависимостей от составного ключа. Если первичный ключ составной, каждый неключевой атрибут должен зависеть от всего ключа.

### 3НФ

Нет транзитивных зависимостей. Неключевые поля не должны зависеть друг от друга.

## 6. Ограничения и целостность данных

Реляционные СУБД поддерживают ограничения, которые защищают данные от некорректных изменений.

### Основные ограничения

- **PRIMARY KEY** — уникальный идентификатор строки;
    
- **FOREIGN KEY** — ссылка на строку в другой таблице;
    
- **UNIQUE** — уникальность значения;
    
- **NOT NULL** — запрет пустых значений;
    
- **CHECK** — пользовательское условие.
    

Пример:

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    email TEXT UNIQUE NOT NULL,
    age INT CHECK (age >= 0)
);
```

## 7. Транзакции и свойства ACID

Одна из главных сильных сторон реляционных СУБД — поддержка транзакций.

### Транзакция

Это логически единый набор операций, который выполняется целиком либо не выполняется вообще.

Пример: перевод денег между счетами.

### Свойства ACID

- **Atomicity** — атомарность. Все операции транзакции выполняются как единое целое.
    
- **Consistency** — согласованность. Транзакция переводит БД из одного корректного состояния в другое.
    
- **Isolation** — изолированность. Параллельные транзакции не должны ломать друг другу результат.
    
- **Durability** — долговечность. После фиксации данные не должны теряться даже при сбое.
    

Именно из-за ACID реляционные БД очень удобны для банковских, учетных и ERP-систем.

## 8. Почему реляционные БД не всегда удобны для распределения

Реляционные БД отлично работают на одном мощном сервере, но при росте нагрузки возникают сложности.

### Основные проблемы горизонтального масштабирования

- join между данными на разных узлах дорогой;
    
- транзакции между несколькими узлами сложны;
    
- глобальные блокировки и согласование тормозят систему;
    
- сложнее обеспечивать строгую консистентность в распределенной среде.
    

Поэтому классические реляционные СУБД долгое время в основном масштабировались **вертикально**:

- больше CPU;
    
- больше RAM;
    
- быстрее диски.
    

---

## 9. Причины появления NoSQL

Когда появились системы с огромными объемами данных и очень высокой нагрузкой, стало ясно, что не для всех задач нужен полный набор возможностей реляционной модели.

NoSQL-системы появились как ответ на задачи:

- горизонтального масштабирования;
    
- работы на кластере;
    
- высокой доступности;
    
- гибкой схемы;
    
- быстрой записи и чтения на большом количестве узлов.
    

Важно: NoSQL не значит “без SQL вообще”. Чаще это означает “не только реляционный подход”.

## 10. Основные классы NoSQL-систем

## 10.1. Key-Value хранилища

### Идея

Данные хранятся как пары:

- ключ;
    
- значение.
    

Пример:

```text
user:123 -> {"name":"Ivan","age":25}
```

### Плюсы

- очень быстрое чтение по ключу;
    
- просто распределять по узлам;
    
- хорошо подходит для кэшей, сессий, профилей, счетчиков.
    

### Минусы

- система почти ничего не знает о внутренней структуре значения;
    
- трудно делать сложные выборки по полям внутри значения;
    
- почти нет join и богатой аналитики.
    

### Где применяют

- Redis;
    
- Riak;
    
- Dynamo-подобные системы;
    
- распределенные кэши.
    

---

## 10.2. Документо-ориентированные БД

### Идея

Значение хранится как документ, обычно JSON-подобный.

Пример документа:

```json
{
  "id": 1,
  "name": "Ivan",
  "emails": ["a@mail.com", "b@mail.com"],
  "address": {
    "city": "Moscow",
    "street": "Lenina"
  }
}
```

### Особенности

- схема может быть гибкой;
    
- документы внутри одной коллекции могут иметь разную структуру;
    
- удобно хранить иерархические данные;
    
- можно индексировать поля документа.
    

### Плюсы

- удобно для быстро меняющейся схемы;
    
- естественно хранить вложенные объекты;
    
- проще разрабатывать приложения, где объектная модель близка к JSON.
    

### Минусы

- меньше строгости, чем в реляционных БД;
    
- сложные междокументные связи неудобнее;
    
- при сильной связанности данных реляционная модель часто лучше.
    

### Примеры

- MongoDB;
    
- Couchbase;
    
- CouchDB.
    

---

## 10.3. Wide Column / Column Family

### Идея

Данные логически организованы по ключу и по колонкам, сгруппированным в семейства. Часто данные распределяются по кластеру по одному ключу и упорядочиваются внутри партиции по другому.

На примере Cassandra обычно различают:

- **partition key** — определяет, на каком узле лежат данные;
    
- **clustering key** — определяет порядок данных внутри партиции.
    

### Преимущества

- хорошо подходит для огромных объемов данных;
    
- эффективна запись;
    
- удобно хранить временные ряды, события, историю действий.
    

### Типичный кейс

Все события пользователя за день или все заказы клиента по времени.

### Минусы

- запросы должны быть заранее продуманы под модель хранения;
    
- join почти отсутствуют;
    
- модель проектируется не “от сущностей”, а “от шаблонов чтения”.
    

### Примеры

- Cassandra;
    
- HBase;
    
- ScyllaDB.
    

---

## 10.4. Графовые БД

### Идея

Данные хранятся как граф:

- **вершины** — объекты;
    
- **ребра** — связи;
    
- и у вершин, и у ребер могут быть свойства.
    

Пример:

- пользователь A дружит с пользователем B;
    
- пользователь B работает в компании C.
    

### Когда они особенно полезны

- социальные графы;
    
- рекомендательные системы;
    
- антифрод;
    
- маршруты и цепочки поставок;
    
- поиск сложных зависимостей.
    

### Плюсы

- очень удобны для обхода связей;
    
- естественно моделируют отношения “многие ко многим”;
    
- хорошо подходят для запросов вида “найти путь”, “друзья друзей”, “все связи на 3 шага”.
    

### Минусы

- не всегда лучший выбор для обычного OLTP;
    
- для простых табличных данных избыточны.
    

### Примеры

- Neo4j;
    
- JanusGraph;
    
- ArangoDB.
    

---

## 11. Отдельно: column-oriented аналитические СУБД

Это не совсем тот же класс, что wide column NoSQL.

Здесь данные физически хранятся **по столбцам**, а не по строкам. Это особенно эффективно для аналитических запросов, когда нужно читать немного колонок, но очень много строк.

### Плюсы

- высокая скорость аналитики;
    
- хорошее сжатие;
    
- удобно для BI, отчетности, витрин данных.
    

### Минусы

- хуже подходят для частых точечных обновлений;
    
- не всегда хороши для классического OLTP.
    

### Примеры

- ClickHouse;
    
- Vertica;
    
- Apache Druid;
    
- BigQuery как аналитическая система схожего класса.

## 12. OLTP и OLAP

Это важное разделение, которое помогает понять, какая БД нужна под какую задачу.

### OLTP

Операционные системы.

Характерно:

- много коротких транзакций;
    
- вставка, обновление, удаление;
    
- работа пользователей в реальном времени;
    
- высокие требования к консистентности.
    

Примеры:

- интернет-магазин;
    
- банковское приложение;
    
- CRM;
    
- ERP.
    

Обычно здесь хороши реляционные СУБД.

### OLAP

Аналитические системы.

Характерно:

- чтение больших массивов данных;
    
- сложные агрегаты;
    
- построение отчетов;
    
- исторические данные.
    

Примеры:

- дашборды;
    
- data warehouse;
    
- аналитика продаж;
    
- мониторинг метрик.
    

Обычно здесь хороши колоночные аналитические СУБД.

## 13. Репликация, шардинг и кластеризация

## Репликация

Копирование данных между серверами.

Нужна для:

- отказоустойчивости;
    
- масштабирования чтения;
    
- резервирования.
    

### Виды

- master-replica;
    
- multi-master;
    
- синхронная;
    
- асинхронная.
    

## Шардинг

Разбиение данных по нескольким узлам.

Например:

- пользователи 1–1 000 000 на одном узле;
    
- пользователи 1 000 001–2 000 000 на другом.
    

Или разбиение по хешу ключа.

## Кластеризация

Общий термин для группы серверов, работающих как единая система.

## 14. CAP-теорема и компромиссы распределенных систем

Для распределенной системы нельзя одновременно идеально гарантировать:

- **Consistency** — все узлы видят одинаковые данные в один и тот же момент;
    
- **Availability** — система всегда отвечает;
    
- **Partition tolerance** — система продолжает работать при сетевых разделениях.
    

На практике при сетевых проблемах приходится выбирать компромисс между консистентностью и доступностью.

### Отсюда возникают разные модели согласованности

- строгая консистентность;
    
- eventual consistency;
    
- tunable consistency;
    
- quorum-based reading/writing.
    

Эти вопросы особенно важны для NoSQL и распределенных систем.

## 15. Сравнение реляционных и NoSQL систем

## Реляционные БД

### Сильные стороны

- строгая схема;
    
- ACID-транзакции;
    
- удобные join;
    
- мощный SQL;
    
- зрелый оптимизатор;
    
- хороши для сложной бизнес-логики.
    

### Слабые стороны

- тяжелее горизонтально масштабировать;
    
- схема более жесткая;
    
- сложнее работать с очень разнородными и быстро меняющимися структурами данных.
    

## NoSQL

### Сильные стороны

- проще распределять по кластеру;
    
- высокая масштабируемость;
    
- высокая доступность;
    
- гибкая схема;
    
- часто лучше подходят для конкретных высоконагруженных сценариев.
    

### Слабые стороны

- слабее поддержка join;
    
- меньше универсальности;
    
- нередко приходится подстраивать модель данных под шаблоны запросов;
    
- меньше строгих гарантий консистентности в ряде систем.

## 16. Когда какой тип БД выбирать

### Реляционная БД подходит, если

- данные сильно связаны;
    
- нужны join;
    
- нужна строгая целостность;
    
- важны транзакции;
    
- предметная область хорошо описывается таблицами.
    

### Key-Value подходит, если

- доступ почти всегда по ключу;
    
- нужен очень быстрый кэш;
    
- данные простые по структуре.
    

### Документная БД подходит, если

- данные естественно выглядят как JSON;
    
- схема часто меняется;
    
- удобно хранить вложенные структуры.
    

### Wide Column подходит, если

- очень большие объемы;
    
- данные распределены по времени или ключам;
    
- запросы заранее известны;
    
- нужна высокая масштабируемость записи.
    

### Графовая БД подходит, если

- главная ценность — связи между объектами;
    
- нужны обходы графа, поиск путей, рекомендации.
    

### Колоночная аналитическая БД подходит, если

- приоритет — отчеты и аналитика;
    
- большие объемы чтения и агрегации;
    
- данные чаще читаются, чем изменяются.

## 17. NewSQL и современные гибридные подходы

Появился класс систем, которые пытаются сочетать:

- SQL;
    
- транзакционность;
    
- горизонтальное масштабирование.
    

Их иногда относят к **NewSQL**.

### Идея

Сохранить удобство реляционной модели, но лучше работать в распределенной среде.

### Примеры

- CockroachDB;
    
- Google Spanner;
    
- YugabyteDB.
    

Также современные реляционные СУБД сами стали поддерживать многое из мира NoSQL:

- JSON и JSONB;
    
- полнотекстовый поиск;
    
- расширяемые типы индексов;
    
- распределенные расширения;
    
- партиционирование.

## 19. Как эволюционировал мир БД

Упрощенно развитие можно представить так:

1. Хранение в файлах.
    
2. Ранние иерархические и сетевые БД.
    
3. Реляционная революция и SQL.
    
4. Рост OLTP-систем и корпоративных СУБД.
    
5. Масштабирование интернета и появление NoSQL.
    
6. Развитие распределенных, аналитических и специализированных систем.
    
7. Гибридный этап: SQL + JSON + распределение + аналитика в разных комбинациях.
    

То есть новые типы БД не “заменили” реляционные, а дополнили их. Сейчас обычно выбирают не “лучшую вообще”, а “лучшую под конкретный класс задач”.

## 20. Краткий итог

- Реляционные БД появились благодаря идее отделить логику работы с данными от их физического хранения.
    
- Их основа — таблицы, SQL, ограничения целостности и транзакции ACID.
    
- Они отлично подходят для бизнес-приложений, где важны связи, точность и консистентность.
    
- NoSQL-системы появились как ответ на задачи распределения, масштабирования и гибкости схемы.
    
- Основные классы NoSQL: key-value, документные, wide column, графовые.
    
- Отдельно важны аналитические колоночные БД для OLAP-задач.
    
- На практике выбор БД зависит от характера данных, шаблонов запросов, требований к консистентности, доступности и масштабируемости.

## 21. Короткая памятка

**Реляционная БД** — когда нужны таблицы, SQL, join, ACID.  
**KV** — когда нужен очень быстрый доступ по ключу.  
**Документная** — когда данные похожи на JSON и схема часто меняется.  
**Wide Column** — когда огромные объемы и запросы заранее известны.  
**Графовая** — когда главное это связи.  
**Колоночная аналитическая** — когда нужны агрегаты и отчеты по большим данным.
