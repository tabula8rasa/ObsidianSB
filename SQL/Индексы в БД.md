## Что такое индекс

**Индекс** - это вспомогательная структура данных, которая позволяет СУБД находить строки не полным перебором таблицы, а по более короткому пути.

Индекс обычно хранит:

- **ключ** - значение индексируемого столбца или набора столбцов
- **ссылку на строку** - адрес строки, RID, ROWID, TID или привязку к primary key, в зависимости от СУБД

---

## Зачем нужны индексы

Индексы нужны, чтобы ускорять:

- `WHERE`
- `JOIN`
- `ORDER BY`
- `GROUP BY`
- `DISTINCT`
- диапазонные выборки
- поиск по префиксу
- иногда агрегаты `MIN`, `MAX`

Но за ускорение чтения мы платим:

- дополнительным местом на диске
- удорожанием `INSERT`
- удорожанием `UPDATE`
- удорожанием `DELETE`
- обслуживанием индексов
- риском неправильного выбора индекса

---

## Как индекс устроен физически

В общем виде индекс - это отдельная структура, отличная от таблицы.

Обычно таблица и индекс лежат отдельно:

- таблица хранит сами строки
- индекс хранит ключи и указатели на строки

Исключение - **кластерные индексы**, где сами данные таблицы физически организованы по ключу индекса.

---

## Как СУБД ищет данные с индексом и без

### Без индекса

СУБД делает **Full Table Scan / Seq Scan**:

1. читает блоки таблицы последовательно
2. проверяет каждую строку на условие
3. возвращает совпадения

Это может быть нормально, если:

- таблица маленькая
- нужно прочитать почти всю таблицу
- условие не селективно

### С индексом

СУБД:

1. ищет значение в индексе
2. получает ссылки на строки
3. извлекает сами строки из таблицы

Если подходящих строк мало, это очень выгодно.

Если строк слишком много, переходы "индекс -> таблица -> индекс -> таблица" могут стать дороже, чем последовательное чтение таблицы целиком.

## B-tree - основной индекс в реляционных БД

Это самый важный и самый часто используемый тип индекса.

### Где применяется

Подходит почти для всего:

- `=`
- `>`
- `<`
- `>=`
- `<=`
- `BETWEEN`
- `IN`
- `ORDER BY`
- `MIN/MAX`
- `LIKE 'abc%'`

### Структура

B-tree состоит из:

- корня
- промежуточных узлов
- листьев

В листьях значения хранятся в **отсортированном порядке**.

Обычно высота дерева невелика - 3-4 уровня даже на очень больших таблицах.

### Как идет поиск

Допустим, есть индекс по `salary`.

Чтобы найти `salary = 100000`, СУБД:

1. смотрит в корень
2. понимает, в какой диапазон попадает 100000
3. спускается ниже
4. доходит до листа
5. находит нужное значение и ссылки на строки

### Почему B-tree так хорош

Потому что:

- поиск быстрый
- диапазоны быстрые
- сортировка уже есть
- структура сбалансирована

### Где B-tree работает плохо

- по низкоселективным колонкам типа `gender`
- по шаблонам `LIKE '%abc'`
- когда почти вся таблица попадает под условие
- когда выражение в `WHERE` не совпадает с индексом
- когда на колонке часто меняются значения

---

## Bitmap индекс

### Идея

Для каждого уникального значения хранится битовая карта:

- 1 - строка содержит это значение
- 0 - не содержит

Пример: колонка `gender`

| row | gender |
|---|---|
| 1 | M |
| 2 | F |
| 3 | M |
| 4 | F |

Bitmap для `M`:

`1010`

Bitmap для `F`:

`0101`

### Когда хорош

- мало уникальных значений
- аналитические запросы
- много условий `AND`, `OR`
- хранилища данных
- OLAP

### Когда плох

- частые обновления
- OLTP
- интенсивные `INSERT/UPDATE/DELETE`

### Почему

Обновление одной строки может потребовать модифицировать целые битовые сегменты. Это создает блокировки и дорого стоит.

---
## Индекс по выражениюw
```sql
create index on table_name((id + 3));
```
Выражение вычисляется один раз и хранится в индексе. При обращении по условию 
```sql
where id + 3 = 5
```
будет использоваться данный индекс

## Hash индекс

### Идея

Хешируется значение ключа и по хешу ищется запись.

### Хорошо работает для

- точного равенства `=`

### Плохо работает для

- диапазонов
- сортировок
- `BETWEEN`
- `LIKE 'abc%'`

### Пример

Хорошо:

```sql
WHERE id = 100
````

Плохо:

```sql
WHERE id > 100
ORDER BY id
```

Эти типы особенно важны в PostgreSQL.

## GIN
GIN расшифровывается как «Generalized Inverted Index» (Обобщённый инвертированный индекс). GIN предназначается для случаев, когда индексируемые значения являются составными, а запросы, на обработку которых рассчитан индекс, ищут значения элементов в этих составных объектах. Например, такими объектами могут быть документы, а запросы могут выполнять поиск документов, содержащих определённые слова.

Здесь мы используем термин _объект_, говоря о составном значении, которое индексируется, и термин _ключ_, говоря о включённом в него элементе. GIN всегда хранит и ищет ключи, а не объекты как таковые.

Индекс GIN сохраняет набор пар (ключ, список идентификаторов), где _список идентификаторов_ содержит идентификаторы строк, в которых находится ключ. Один и тот же идентификатор строки может фигурировать в нескольких списках, так как объект может содержать больше одного ключа. Значение каждого ключа хранится только один раз, так что индекс GIN очень компактен в случаях, когда один ключ встречается много раз.

GIN является обобщённым в том смысле, что код метода доступа GIN не должен знать о конкретных операциях, которые он ускоряет. Вместо этого задаются специальные стратегии для конкретных типов данных. Стратегия определяет, как извлекаются ключи из индексируемых объектов и условий запросов, и как установить, действительно ли удовлетворяет запросу строка, содержащая некоторые значения ключей.

Ключевым преимуществом GIN является то, что он позволяет разрабатывать дополнительные типы данных с соответствующими методами доступа экспертам в предметной области типа данных, а не специалистам по СУБД. В этом аспекте он похож на GiST.

GIN полезен для:

- массивов
    
- JSONB
    
- полнотекстового поиска
    
- `tsvector`
    
- операций "содержит"
    

### Пример

```sql
CREATE INDEX idx_docs_tags_gin ON documents USING gin(tags);
```

Запрос:

```sql
SELECT *
FROM documents
WHERE tags @> ARRAY['postgres'];
```

---

## GiST

GiST - более общий индексный каркас.

Используется для:

- геоданных
    
- диапазонов
    
- полнотекста
    
- пользовательских операторов
    

### Пример

```sql
CREATE INDEX idx_rooms_period_gist
ON bookings USING gist(period);
```

---

## BRIN

BRIN - очень компактный индекс для очень больших таблиц, где значения коррелируют с физическим порядком хранения.

Хорош для:

- временных рядов
    
- логов
    
- событий
    
- append-only таблиц
    

### Идея

BRIN не индексирует каждую строку. Он хранит сводную информацию о диапазонах страниц.

### Когда выгоден

Если данные записываются примерно в порядке времени:

```sql
created_at
2026-01-01
2026-01-01
2026-01-02
2026-01-02
...
```

### Пример

```sql
CREATE INDEX idx_events_created_at_brin
ON events USING brin(created_at);
```

## Кластерный и некластерный индекс

## Кластерный индекс

Кластерный индекс означает, что **данные таблицы физически упорядочены по ключу индекса**.

### Следствия

- строки с близкими ключами лежат рядом
    
- диапазонные выборки быстрые
    
- обычно на таблицу только один кластерный индекс
    

### Пример из MySQL InnoDB

В InnoDB `PRIMARY KEY` - это кластерный индекс.

То есть данные таблицы физически лежат по PK.

---

## Некластерный индекс

Некластерный индекс хранит:

- значение ключа
    
- указатель на строку
    

Таблица живет отдельно и может быть физически организована как угодно.

На одну таблицу может быть много некластерных индексов.

---

## Уникальный, составной, частичный, покрывающий, функциональный индекс

## Уникальный индекс

Гарантирует, что одинаковых ключей не будет.

```sql
CREATE UNIQUE INDEX idx_users_email ON users(email);
```

Подходит для:

- email
    
- логинов
    
- внешних бизнес-идентификаторов
    

---

## Составной индекс

Индекс по нескольким столбцам.

```sql
CREATE INDEX idx_orders_user_status_created
ON orders(user_id, status, created_at);
```

Очень важен **порядок столбцов**.

---

## Частичный индекс

Индексируется не вся таблица, а только часть строк.

### Пример

```sql
CREATE INDEX idx_orders_active
ON orders(user_id)
WHERE status = 'active';
```

Плюсы:

- меньше размер
    
- дешевле поддержка
    
- выше эффективность для конкретного сценария
    

---

## Покрывающий индекс

Индекс, который содержит все нужные для запроса поля, чтобы не ходить в таблицу.

### Идея

Если запросу нужны только поля, которые уже есть в индексе, СУБД может выполнить **Index Only Scan**.

### Пример для PostgreSQL

```sql
CREATE INDEX idx_orders_user_created_include
ON orders(user_id, created_at)
INCLUDE (amount, status);
```

Запрос:

```sql
SELECT created_at, amount, status
FROM orders
WHERE user_id = 10;
```

---

## Функциональный индекс

Индекс строится не по самому столбцу, а по выражению.

### Пример

```sql
CREATE INDEX idx_users_lower_email
ON users(LOWER(email));
```

Тогда такой запрос сможет использовать индекс:

```sql
SELECT *
FROM users
WHERE LOWER(email) = 'test@example.com';
```

А такой запрос, скорее всего, нет:

```sql
SELECT *
FROM users
WHERE email = 'test@example.com';
```

---

## Descending индекс

Помогает для сортировки по убыванию.

```sql
CREATE INDEX idx_orders_created_desc
ON orders(created_at DESC);
```

---

## Reverse key index

Обычно встречается в Oracle.

Используется, чтобы снизить конкуренцию за правый край B-tree при монотонно растущих ключах типа sequence.

---

## Селективность, кардинальность, factor кластеризации

## Кардинальность

Количество уникальных значений в столбце.

Примеры:

- `id` - высокая кардинальность
    
- `gender` - низкая кардинальность
    
- `status` из 5 значений - низкая кардинальность
    
- `email` - очень высокая кардинальность
    

---

## Селективность

Насколько условие отбирает малую часть данных.

Высокоселективное условие:

```sql
WHERE email = 'a@b.com'
```

Низкоселективное условие:

```sql
WHERE status = 'active'
```

если `active` у 90% строк.

---

## Factor кластеризации

Это степень соответствия между порядком строк в таблице и порядком ключей в индексе.

Если строки на диске лежат примерно так же, как в индексе, диапазонное чтение очень эффективно.

Если строки разбросаны случайно, СУБД придется делать много случайных чтений.

---

## Когда индекс работает хорошо

Индекс хорош, когда:

- таблица большая
    
- условие селективно
    
- запрос возвращает малую часть строк
    
- колонка часто участвует в `WHERE`
    
- колонка участвует в `JOIN`
    
- колонка участвует в `ORDER BY`
    
- используется диапазон
    
- порядок столбцов в составном индексе выбран правильно
    
- статистика актуальна
    
- нет лишних преобразований в условии
    

---

## Когда индекс не помогает или даже мешает

Индекс может не помочь, если:

- таблица маленькая
    
- выбирается большая доля строк
    
- на колонке низкая селективность
    
- запрос оборачивает столбец функцией
    
- есть неудачный шаблон `LIKE '%abc'`
    
- типы данных не совпадают и идет неявное преобразование
    
- используется слишком много индексов и запись стала дорогой
    
- статистика устарела
    
- индексы раздуты
    
- планировщик считает full scan дешевле
    

---

## Как оптимизатор выбирает план

SQL - декларативный язык. Ты говоришь **что** хочешь, а не **как** это делать.

Решение принимает оптимизатор.

Он учитывает:

- количество строк в таблице
    
- число уникальных значений
    
- распределение значений
    
- стоимость случайного чтения
    
- стоимость последовательного чтения
    
- ожидаемое число строк
    
- наличие сортировки
    
- наличие join
    
- доступные индексы
    
- статистику

## Как читать EXPLAIN

Ниже - самые важные узлы плана.

### Seq Scan

Полное сканирование таблицы.

```sql
EXPLAIN SELECT * FROM users WHERE age = 30;
```

Пример:

```text
Seq Scan on users  (cost=0.00..4312.00 rows=100 width=64)
  Filter: (age = 30)
```

Это значит:

- таблица читается целиком
    
- потом к строкам применяется фильтр
    

---

### Index Scan

Поиск через индекс, затем чтение строк из таблицы.

```text
Index Scan using idx_users_age on users  (cost=0.29..121.00 rows=100 width=64)
  Index Cond: (age = 30)
```

---

### Index Only Scan

Чтение только из индекса, без захода в таблицу.

```text
Index Only Scan using idx_users_age_include on users
  Index Cond: (age = 30)
```

Очень хорошо, но в PostgreSQL возможно не всегда из-за MVCC и visibility map.

---

### Bitmap Index Scan + Bitmap Heap Scan

Оптимально, когда строк довольно много, но не почти вся таблица.

```text
Bitmap Heap Scan on users
  Recheck Cond: (status = 'active')
  -> Bitmap Index Scan on idx_users_status
       Index Cond: (status = 'active')
```

Идея:

1. собрать bitmap по совпавшим строкам
    
2. читать таблицу уже пачками, а не по одной строке
    

---

### Sort

Если индекс не смог обеспечить нужный порядок, СУБД будет сортировать отдельно.

---

### Hash Join / Merge Join / Nested Loop

Индексы очень важны для join, особенно для nested loop и merge join.

---

## Практические SQL примеры

---

## Пример 1. Индекс для поиска пользователя по email

```sql
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    email TEXT NOT NULL,
    full_name TEXT,
    created_at TIMESTAMP NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX idx_users_email ON users(email);
```

Запрос:

```sql
SELECT *
FROM users
WHERE email = 'ivan@example.com';
```

Подходит идеально:

- высокая кардинальность
    
- точный поиск
    
- возвращает 1 строку
    

---

## Пример 2. Поиск по статусу не всегда выгоден

```sql
CREATE TABLE orders (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL,
    status TEXT NOT NULL,
    amount NUMERIC(12,2),
    created_at TIMESTAMP NOT NULL
);

CREATE INDEX idx_orders_status ON orders(status);
```

Запрос:

```sql
SELECT *
FROM orders
WHERE status = 'paid';
```

Если `paid` у 85% заказов, индекс может не использоваться, потому что дешевле прочитать таблицу целиком.

---

## Пример 3. Составной индекс

```sql
CREATE INDEX idx_orders_user_status_created
ON orders(user_id, status, created_at);
```

Хорошо для:

```sql
SELECT *
FROM orders
WHERE user_id = 10
  AND status = 'paid'
ORDER BY created_at;
```

---

## Пример 4. Функциональный индекс

```sql
CREATE INDEX idx_users_lower_email
ON users(LOWER(email));
```

Запрос:

```sql
SELECT *
FROM users
WHERE LOWER(email) = 'ivan@example.com';
```

---

## Пример 5. Частичный индекс

```sql
CREATE INDEX idx_orders_open_user
ON orders(user_id)
WHERE status IN ('new', 'processing');
```

Подходит для системы, где часто открывают только активные заказы.

---

## Пример 6. Покрывающий индекс

```sql
CREATE INDEX idx_orders_user_created_include
ON orders(user_id, created_at DESC)
INCLUDE (status, amount);
```

Запрос:

```sql
SELECT created_at, status, amount
FROM orders
WHERE user_id = 10
ORDER BY created_at DESC
LIMIT 20;
```

Индекс очень хорош, потому что:

- фильтрация по `user_id`
    
- сортировка по `created_at DESC`
    
- остальные поля можно взять из индекса
    

---

## Составные индексы и правило левого префикса

Это один из самых важных разделов.

Пусть есть индекс:

```sql
CREATE INDEX idx_t_a_b_c ON t(a, b, c);
```

### Обычно индекс хорошо работает для:

```sql
WHERE a = ...
WHERE a = ... AND b = ...
WHERE a = ... AND b = ... AND c = ...
WHERE a = ... ORDER BY b
WHERE a = ... AND b > ...
```

### Обычно не работает полноценно для:

```sql
WHERE b = ...
WHERE c = ...
WHERE b = ... AND c = ...
```

Потому что отсутствует левый префикс `a`.

---

## Важное правило

**Составной индекс используется слева направо.**

Поэтому порядок столбцов надо выбирать под реальные запросы.

---

## Как выбирать порядок в составном индексе

Обычно раньше ставят:

1. столбцы, по которым часто идет точное равенство
    
2. затем столбцы диапазона
    
3. затем столбцы сортировки
    
4. затем include-поля
    

### Пример

Есть запрос:

```sql
SELECT *
FROM orders
WHERE user_id = 10
  AND status = 'paid'
  AND created_at >= '2026-01-01'
ORDER BY created_at DESC;
```

Часто хороший индекс:

```sql
(user_id, status, created_at DESC)
```

---

## LIKE, ILIKE, поиск по тексту и индексы

## B-tree и LIKE

Индекс B-tree обычно может использоваться для:

```sql
WHERE name LIKE 'Иван%'
```

Потому что известен левый край строки.

### Но не для

```sql
WHERE name LIKE '%ван'
WHERE name LIKE '%ва%'
```

Потому что префикс неизвестен.

---

## ILIKE

В PostgreSQL обычный B-tree для `ILIKE` часто не помогает так, как хочется.

Решение:

- функциональный индекс `LOWER(column)`
    
- trigram индекс
    
- полнотекстовый поиск
    

### Пример

```sql
CREATE INDEX idx_users_lower_name
ON users(LOWER(name));
```

Запрос:

```sql
SELECT *
FROM users
WHERE LOWER(name) LIKE 'иван%';
```

---

## Trigram индекс

В PostgreSQL для поиска по подстроке удобно использовать `pg_trgm`.

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX idx_users_name_trgm
ON users USING gin(name gin_trgm_ops);
```

Теперь запросы типа:

```sql
SELECT *
FROM users
WHERE name ILIKE '%ван%';
```

могут работать заметно лучше.

---

## Полнотекстовый поиск

Для больших текстов лучше не `LIKE`, а FTS.

```sql
CREATE INDEX idx_docs_fts
ON documents USING gin(to_tsvector('russian', body));
```

Запрос:

```sql
SELECT *
FROM documents
WHERE to_tsvector('russian', body) @@ plainto_tsquery('russian', 'индексы базы данных');
```

---

## JOIN и индексы

Индексы очень важны в join.

### Классический случай

```sql
SELECT o.id, u.email
FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.created_at >= '2026-01-01';
```

### Какие индексы полезны

```sql
users(id)         -- обычно PK уже есть
orders(user_id)
orders(created_at)
```

Если join идет часто вместе с фильтром по дате, может подойти:

```sql
CREATE INDEX idx_orders_created_user
ON orders(created_at, user_id);
```

или

```sql
CREATE INDEX idx_orders_user_created
ON orders(user_id, created_at);
```

Какой лучше - зависит от реальных запросов.

---

## Что важно помнить про JOIN

- на PK индекс обычно уже есть
    
- на FK индекс не всегда создается автоматически
    
- если по внешнему ключу часто делают join, индекс на FK очень желателен
    

---

## ORDER BY, GROUP BY, DISTINCT и индексы

## ORDER BY

Если индекс совпадает с условием и порядком сортировки, отдельная сортировка может не понадобиться.

### Пример

```sql
CREATE INDEX idx_orders_user_created_desc
ON orders(user_id, created_at DESC);
```

Запрос:

```sql
SELECT *
FROM orders
WHERE user_id = 10
ORDER BY created_at DESC
LIMIT 50;
```

Очень хороший сценарий для индекса.

---

## GROUP BY

Индекс может помочь, особенно если данные уже идут в нужном порядке.

### Пример

```sql
SELECT user_id, COUNT(*)
FROM orders
GROUP BY user_id;
```

Индекс по `user_id` может быть полезен, но не всегда. Все зависит от объема и плана.

---

## DISTINCT

```sql
SELECT DISTINCT status
FROM orders;
```

Индекс по `status` может помочь, особенно если столбец короткий и таблица большая.

---

## Индексы и NULL

Это тонкая тема, и она зависит от СУБД.

### Oracle

Обычный B-tree индекс обычно **не хранит записи, где все ключевые поля NULL**.

Поэтому:

```sql
WHERE column IS NULL
```

может не использовать такой индекс.

### PostgreSQL

PostgreSQL умеет индексировать NULL в B-tree, и запрос `IS NULL` может использовать индекс.

### MySQL

Поведение отличается от Oracle и чаще ближе к ожидаемому.

---

## Практическое правило

Если часто нужен поиск по `IS NULL` или `IS NOT NULL`, это надо проверять на конкретной СУБД и на реальном плане.

---

## Индексы и операции INSERT UPDATE DELETE

Индекс ускоряет чтение, но замедляет запись.

### INSERT

При вставке строки надо:

- вставить строку в таблицу
    
- вставить ключи во все связанные индексы
    

### UPDATE

Если меняется индексируемый столбец, часто это почти как:

- удалить старое значение из индекса
    
- вставить новое
    

### DELETE

Нужно удалить запись и из таблицы, и из всех индексов.

---

## Почему много индексов - плохо

Если на таблице 10 индексов, каждый `INSERT` и `UPDATE` становится дороже.

Поэтому индекс - это не бесплатное ускорение, а компромисс.

---

## Партиционирование и индексы

Партиционирование - это разделение большой таблицы на части.

### Зачем нужно

- упрощает работу с огромными таблицами
    
- помогает запросам по датам
    
- облегчает удаление старых данных
    
- улучшает обслуживание
    

### Типы

- Range
    
- List
    
- Hash
    

### Пример

Таблица событий по месяцам.

Если запрос:

```sql
WHERE created_at >= '2026-03-01'
  AND created_at < '2026-04-01'
```

то СУБД может читать только нужную партицию.

### Важно

Партиционирование не заменяет индексы, а дополняет их.

---

## Особенности PostgreSQL

Этот раздел очень полезен для практики.

### Какие индексы есть

- B-tree
    
- Hash
    
- GiST
    
- SP-GiST
    
- GIN
    
- BRIN
    

### PostgreSQL и MVCC

Даже если все нужные поля есть в индексе, PostgreSQL не всегда может сделать `Index Only Scan`.

Причина - MVCC. Нужно убедиться, что строка видима, а для этого используется **visibility map**.

### INCLUDE

Очень полезно для покрывающих индексов.

```sql
CREATE INDEX idx_orders_user_include
ON orders(user_id)
INCLUDE (amount, status);
```

### Частичные индексы

Очень сильная сторона PostgreSQL.

```sql
CREATE INDEX idx_active_users
ON users(id)
WHERE deleted_at IS NULL;
```

### Выражения

```sql
CREATE INDEX idx_lower_email
ON users(LOWER(email));
```

### BRIN

Очень хорош на больших append-only таблицах.

### pg_trgm

Полезно для `LIKE '%abc%'`, `ILIKE`, fuzzy search.

### EXPLAIN ANALYZE

Очень важный инструмент.

```sql
EXPLAIN ANALYZE
SELECT *
FROM orders
WHERE user_id = 10;
```

Смотри на:

- actual time
    
- rows
    
- loops
    
- отличаются ли estimate и actual
    
- есть ли sort
    
- есть ли seq scan
    
- есть ли nested loop на большом объеме

## Полезные команды PostgreSQL

### Посмотреть индексы таблицы

```sql
\d orders
```

или

```sql
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'orders';
```

### Размер индексов

```sql
SELECT
    relname AS index_name,
    pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
JOIN pg_index USING (indexrelid)
JOIN pg_class ON pg_class.oid = indexrelid
WHERE schemaname = 'public';
```

### Статистика использования индексов

```sql
SELECT
    relname AS table_name,
    indexrelname AS index_name,
    idx_scan,
    idx_tup_read,
    idx_tup_fetch
FROM pg_stat_user_indexes
ORDER BY idx_scan DESC;
```

Если индекс не используется вообще, возможно, он лишний.

## Особенности MySQL InnoDB

### Primary Key - кластерный

В InnoDB таблица физически организована по primary key.

### Вторичные индексы

Во вторичном индексе хранится:

- значение вторичного ключа
    
- значение primary key
    

То есть при поиске по вторичному индексу MySQL часто делает:

1. поиск по secondary index
    
2. затем lookup по clustered PK
    

### Следствие

Чем длиннее PK, тем тяжелее вторичные индексы.

Поэтому в InnoDB часто выгодно иметь короткий PK.

### Prefix index

MySQL умеет индексировать префикс строки.

```sql
CREATE INDEX idx_email_prefix ON users(email(20));
```

Это экономит место, но надо аккуратно оценивать селективность.

### EXPLAIN в MySQL

```sql
EXPLAIN SELECT * FROM users WHERE email = 'a@b.com';
```

Смотри на:

- `type`
    
- `possible_keys`
    
- `key`
    
- `rows`
    
- `Extra`
    

Хорошие признаки:

- `const`
    
- `ref`
    
- `range`
    

Плохой признак:

- `ALL` - полное сканирование таблицы

## Особенности Oracle

### B-tree и Bitmap

Oracle широко использует и B-tree, и bitmap.

### Bitmap хорош в DWH

Для аналитики bitmap очень хорош.

### Reverse key

Нужен в сценариях, где много вставок по растущему ключу и идет горячая точка в правой части индекса.

### Function-based index

Очень распространен:

```sql
CREATE INDEX idx_users_upper_name ON users(UPPER(name));
```

### NULL в B-tree

Классический важный вопрос: обычный B-tree индекс Oracle не индексирует полностью NULL ключи.

### Explain plan

Используют:

```sql
EXPLAIN PLAN FOR
SELECT * FROM users WHERE id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

## Типичные ошибки при работе с индексами

### 1. Создать индекс на каждую колонку подряд

Плохо, потому что:

- запись замедляется
    
- место расходуется
    
- optimizer все равно выберет только нужные
    
- лишние индексы надо обслуживать
    

---

### 2. Игнорировать реальные запросы

Индекс делают не "на колонку", а **под типовой запрос**.

Не вопрос:

- "Нужен ли индекс на `created_at`?"
    

А вопрос:

- "Какие запросы по `created_at` выполняются?"
    

---

### 3. Неправильный порядок в составном индексе

Плохой пример:

```sql
(status, user_id)
```

если почти все запросы начинаются с `WHERE user_id = ?`.

---

### 4. Индексировать низкоселективные колонки без понимания

Например:

```sql
status
gender
is_deleted
active_flag
```

Иногда это полезно, но очень часто нет.

---

### 5. Оборачивать колонку функцией и ломать индекс

```sql
WHERE DATE(created_at) = '2026-03-27'
```

Часто ломает возможность использовать обычный индекс по `created_at`.

Лучше так:

```sql
WHERE created_at >= '2026-03-27'
  AND created_at <  '2026-03-28'
```

---

### 6. Игнорировать неявные преобразования типов

Например, колонка `id BIGINT`, а в запросе передают строку.

Это может сломать оптимальный план.

---

### 7. Не обновлять статистику

Если статистика плохая, оптимизатор может выбрать плохой план.

---

### 8. Не проверять EXPLAIN ANALYZE

Нельзя оценивать индекс только "на глаз".

---

### 9. Не индексировать внешние ключи

Если по FK часто идут join или delete/update родительской таблицы, отсутствие индекса может сильно бить по производительности.

---

### 10. Дублировать одинаковые индексы

Например, есть:

```sql
(a)
(a, b)
```

Иногда индекс `(a)` уже частично покрывается `(a, b)`, и отдельный `(a)` лишний. Но это надо проверять по нагрузке и сортировкам.

---

## Как выбрать индекс под задачу

Вот удобный алгоритм.

### Шаг 1. Посмотри на реальный запрос

Нужно понять:

- какие поля в `WHERE`
    
- какие поля в `JOIN`
    
- какие поля в `ORDER BY`
    
- какие поля в `GROUP BY`
    
- какие поля выбираются в `SELECT`
    
- есть ли `LIMIT`
    

---

### Шаг 2. Определи тип условий

- равенство `=`
    
- диапазон `> < BETWEEN`
    
- список `IN`
    
- поиск по префиксу
    
- сортировка
    
- соединение таблиц
    

---

### Шаг 3. Определи селективность

- сколько строк вернет условие
    
- какие значения популярны
    
- есть ли сильный skew распределения
    

---

### Шаг 4. Выбери порядок столбцов

Обычно:

1. условия точного равенства
    
2. затем диапазон
    
3. затем сортировка
    
4. затем include-поля
    

---

### Шаг 5. Проверь, не нужен ли partial index

Если 95% запросов идут только по активным строкам, partial index часто лучше общего.

---

### Шаг 6. Проверь EXPLAIN ANALYZE до и после

Смотри не только на факт использования индекса, но и на реальное ускорение.

---

## Шаблоны решений для рабочих кейсов

---

## Кейc 1. Лента событий по пользователю

Запрос:

```sql
SELECT id, created_at, event_type
FROM events
WHERE user_id = 42
ORDER BY created_at DESC
LIMIT 100;
```

Хороший индекс:

```sql
CREATE INDEX idx_events_user_created_desc
ON events(user_id, created_at DESC)
INCLUDE (event_type);
```

Почему:

- фильтр по `user_id`
    
- сортировка по `created_at DESC`
    
- `LIMIT 100`
    
- `event_type` можно взять из индекса
    

---

## Кейc 2. Поиск активных задач по исполнителю

```sql
SELECT id, title, priority, due_date
FROM tasks
WHERE assignee_id = 100
  AND status IN ('new', 'in_progress')
ORDER BY due_date;
```

Решение:

```sql
CREATE INDEX idx_tasks_active_assignee_due
ON tasks(assignee_id, due_date)
INCLUDE (title, priority)
WHERE status IN ('new', 'in_progress');
```

---

## Кейc 3. Частый join заказов и пользователей

```sql
SELECT o.id, o.amount, u.email
FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.created_at >= now() - interval '7 days';
```

Возможные индексы:

```sql
CREATE INDEX idx_orders_created_user
ON orders(created_at, user_id);

CREATE UNIQUE INDEX idx_users_email ON users(email);
```

PK по `users(id)` уже предполагается.

---

## Кейc 4. Поиск по email без учета регистра

```sql
SELECT *
FROM users
WHERE LOWER(email) = LOWER('Test@Example.com');
```

Решение:

```sql
CREATE INDEX idx_users_lower_email
ON users(LOWER(email));
```

---

## Кейc 5. Логи за период

```sql
SELECT *
FROM logs
WHERE created_at >= '2026-03-01'
  AND created_at <  '2026-03-02';
```

Если таблица огромная и append-only, можно рассмотреть:

```sql
CREATE INDEX idx_logs_created_brin
ON logs USING brin(created_at);
```

Если диапазоны короткие и доступ должен быть очень быстрым, может быть лучше B-tree.

---

## Кейc 6. Поиск по JSONB в PostgreSQL

```sql
SELECT *
FROM orders
WHERE payload @> '{"status":"paid"}';
```

Решение:

```sql
CREATE INDEX idx_orders_payload_gin
ON orders USING gin(payload);
```

---

## Кейc 7. Поиск по подстроке в имени

```sql
SELECT *
FROM customers
WHERE full_name ILIKE '%иван%';
```

Решение в PostgreSQL:

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX idx_customers_name_trgm
ON customers USING gin(full_name gin_trgm_ops);
```

## Практика с EXPLAIN

### До индекса

```sql
EXPLAIN ANALYZE
SELECT *
FROM orders
WHERE user_id = 100
ORDER BY created_at DESC
LIMIT 20;
```

Допустим, видим:

```text
Seq Scan on orders
  Filter: (user_id = 100)
Sort
Limit
```

Проблема:

- таблица читается целиком
    
- потом отдельно сортируется
    

### После индекса

```sql
CREATE INDEX idx_orders_user_created_desc
ON orders(user_id, created_at DESC);
```

Теперь возможен план:

```text
Limit
  -> Index Scan using idx_orders_user_created_desc on orders
       Index Cond: (user_id = 100)
```

Стало лучше, потому что:

- не нужен full scan
    
- не нужна отдельная сортировка
    
- limit отрабатывает быстро
    

---

## Как понять, что индекс действительно помог

Смотри на:

- уменьшился ли `actual time`
    
- уменьшилось ли число прочитанных строк
    
- исчез ли `Sort`
    
- исчез ли `Seq Scan`
    
- стало ли меньше блоков чтения
    
- не стало ли хуже для других важных запросов
    
- не слишком ли выросла стоимость записи

## Индекс и диапазон

Пусть есть индекс:

```sql
(a, b, c)
```

И запрос:

```sql
WHERE a = 10 AND b > 5 AND c = 7
```

После первого диапазона на `b` дальнейшее использование `c` в классическом B-tree уже ограничено.

Это важная идея для собеседования:

**после диапазонного условия эффективность дальнейших колонок индекса снижается**.

---

## Индексы и сортировка

Очень хороший сценарий:

```sql
WHERE a = ?
ORDER BY b
LIMIT 10
```

Индекс:

```sql
(a, b)
```

Потому что данные уже отсортированы по `b` внутри группы `a`.

---

## Индексы и агрегаты MIN/MAX

B-tree индекс особенно полезен для:

```sql
SELECT MIN(created_at) FROM orders;
SELECT MAX(created_at) FROM orders;
```

Или:

```sql
SELECT MAX(created_at)
FROM orders
WHERE user_id = 10;
```

С индексом:

```sql
(user_id, created_at)
```

или

```sql
(user_id, created_at DESC)
```

---

## Индекс только ради уникальности

Иногда индекс создается не ради скорости, а ради ограничения целостности.

```sql
ALTER TABLE users
ADD CONSTRAINT uq_users_email UNIQUE(email);
```

Но такой индекс часто и полезен для запросов.

---

## Индексы на внешние ключи

В некоторых СУБД индекс на FK не создается автоматически.

Пример:

```sql
CREATE TABLE orders (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT REFERENCES users(id)
);
```

Если часто выполняются:

- `JOIN orders ON users`
    
- удаление пользователя
    
- обновление PK родителя
    
- выборка заказов пользователя
    

тогда индекс нужен:

```sql
CREATE INDEX idx_orders_user_id ON orders(user_id);
```

---

## Индексы и массовая загрузка

Если грузишь очень много данных:

- иногда выгодно сначала загрузить данные
    
- потом построить индекс
    

А не вставлять по одной строке в таблицу с уже существующим набором тяжелых индексов.

---

## Индексы и bloat

С течением времени индекс может раздуваться:

- удаленные записи остаются как мусор
    
- обновления создают фрагментацию
    
- pages заполняются неравномерно
    

### Что делать

Зависит от СУБД:

- PostgreSQL: `REINDEX`, `VACUUM`, `VACUUM FULL`, `pg_repack`
    
- Oracle: rebuild index
    
- MySQL: rebuild / optimize / alter table
    

---

## Fillfactor

В некоторых СУБД можно оставить запас места на страницах, чтобы уменьшить split pages при вставках.

Это полезно для индексов, где много вставок в середину диапазона.

---

## Index seek, index scan, full scan - разница в терминах

Термины зависят от СУБД.

### В общем смысле

- **seek** - точечный или узкий поиск по индексу
    
- **index scan** - обход части или всего индекса
    
- **full table scan / seq scan** - чтение всей таблицы
    

---

## Почему optimizer иногда не использует индекс, даже если он есть

Причины:

1. таблица маленькая
    
2. условие не селективно
    
3. статистика устарела
    
4. индекс не покрывает шаблон запроса
    
5. функция в `WHERE`
    
6. неявное приведение типов
    
7. cost model считает full scan дешевле
    
8. нужен почти весь набор строк
    
9. плохой порядок колонок в индексе
    
10. параметризация запроса скрывает реальные значения
    

---

## Parameter sniffing и generic plans

Это уже более продвинутый уровень.

Иногда один и тот же запрос выполняется с разными параметрами:

- для редкого значения индекс очень хорош
    
- для частого значения full scan лучше
    

Но план может кешироваться и использоваться повторно.

Из-за этого "правильный" индекс не всегда дает одинаковый эффект.

---

## Индексы и горячие точки при вставке

Если ключ монотонно растет:

- `id = 1, 2, 3, 4...`
    
- `created_at` только увеличивается
    

то вставки идут в правый край B-tree.

В очень нагруженных системах это может создавать contention.

Решения:

- reverse key index в Oracle
    
- UUID / распределенные ключи
    
- шардирование
    
- настройка fillfactor
    
- снижение write hotspot
    

Но у UUID есть и минусы - хуже locality, больше fragmentation, тяжелее вторичные индексы.

---

## Что лучше - один составной индекс или несколько одиночных

Зависит от запросов.

### Если типичный запрос такой:

```sql
WHERE a = ? AND b = ?
```

то обычно лучше один индекс:

```sql
(a, b)
```

чем два отдельных `(a)` и `(b)`.

### Но

Для некоторых СУБД возможны bitmap/bitmap-like комбинации нескольких индексов.

Однако в OLTP чаще все равно важнее точный составной индекс.

---

## Пример, где два одиночных индекса хуже

Есть запрос:

```sql
SELECT *
FROM orders
WHERE user_id = 10 AND status = 'paid';
```

Индексы:

```sql
(user_id)
(status)
```

Это хуже, чем:

```sql
(user_id, status)
```

если такой паттерн запросов очень частый.

---

## Индексы и UPDATE HOT в PostgreSQL

В PostgreSQL есть HOT update - heap only tuple.

Если обновление не затрагивает индексируемые поля, обновление может быть дешевле, потому что не надо трогать индексные записи.

Следствие:

- не индексируй без необходимости поля, которые часто обновляются

## Вопросы и ответы для собеседования

### Что такое индекс?

Индекс - это дополнительная структура данных, которая ускоряет поиск строк по значениям колонок, но требует места и замедляет операции записи.

---

### Почему индекс ускоряет поиск?

Потому что позволяет не сканировать всю таблицу, а быстро найти указатели на нужные строки через упорядоченную или специальную структуру.

---

### Какие минусы у индекса?

- занимает место
    
- замедляет `INSERT/UPDATE/DELETE`
    
- требует обслуживания
    
- может быть выбран зря
    
- неправильный индекс не помогает
    

---

### Чем кластерный индекс отличается от некластерного?

Кластерный определяет физический порядок данных таблицы. Некластерный хранит ключи отдельно и ссылается на строки таблицы.

---

### Почему на таблицу обычно один кластерный индекс?

Потому что таблицу физически можно отсортировать только одним способом.

---

### Что такое B-tree?

Сбалансированное дерево, в котором ключи хранятся упорядоченно. Это универсальный индекс для равенств, диапазонов, сортировок и префиксного поиска.

---

### Когда индекс может не использоваться?

Когда:

- таблица маленькая
    
- нужно много строк
    
- условие не селективно
    
- статистика плохая
    
- есть функция над колонкой
    
- неправильный тип данных
    
- индекс не подходит под запрос
    

---

### Что такое селективность?

Это степень отсева строк условием. Чем меньше строк подходит, тем выше селективность.

---

### Что такое кардинальность?

Количество уникальных значений в столбце.

---

### Почему порядок колонок в составном индексе важен?

Потому что индекс используется слева направо. Для индекса `(a, b, c)` условия по `a` критичны для эффективного использования `b` и `c`.

---

### Что такое covering index?

Индекс, который содержит все поля, нужные для выполнения запроса, так что чтение таблицы может не понадобиться.

---

### Что такое partial index?

Индекс, построенный только на части строк по условию `WHERE`.

---

### Что такое functional index?

Индекс по выражению, например `LOWER(email)`.

---

### Почему `LIKE '%abc%'` плохо работает с B-tree?

Потому что неизвестен левый префикс строки, и нельзя быстро определить, с какого места начинать поиск в отсортированном дереве.

---

### Почему bitmap индекс хорош для DWH и плох для OLTP?

Потому что bitmap очень эффективен для аналитических фильтров по низкокардинальным полям, но плохо переносит частые изменения данных.

---

### Почему слишком много индексов - плохо?

Потому что каждое изменение строки требует поддерживать каждый индекс.

---

### Нужно ли индексировать внешние ключи?

Во многих практических случаях да, особенно если по ним часто делают join или операции с родительской таблицей.

---

### Чем Index Scan отличается от Index Only Scan?

Index Scan использует индекс, а потом идет в таблицу за строкой. Index Only Scan старается взять все данные прямо из индекса.

---

### Что такое BRIN?

Очень компактный индекс для больших таблиц, где данные физически коррелируют с порядком значений.

---

## Быстрые шаблоны DDL

### Обычный B-tree индекс

```sql
CREATE INDEX idx_name ON table_name(column_name);
```

### Уникальный

```sql
CREATE UNIQUE INDEX idx_name ON table_name(column_name);
```

### Составной

```sql
CREATE INDEX idx_name ON table_name(col1, col2, col3);
```

### Частичный

```sql
CREATE INDEX idx_name ON table_name(col1)
WHERE status = 'active';
```

### Функциональный

```sql
CREATE INDEX idx_name ON table_name(LOWER(email));
```

### PostgreSQL INCLUDE

```sql
CREATE INDEX idx_name ON table_name(col1, col2) INCLUDE (col3, col4);
```

### GIN для JSONB

```sql
CREATE INDEX idx_name ON table_name USING gin(payload);
```

### BRIN

```sql
CREATE INDEX idx_name ON table_name USING brin(created_at);
```

## Пример полноценного разбора задачи

### Задача

Есть таблица заказов 100 млн строк. Частый запрос:

```sql
SELECT id, created_at, amount, status
FROM orders
WHERE user_id = 12345
  AND created_at >= now() - interval '30 days'
ORDER BY created_at DESC
LIMIT 50;
```

### Анализ

- фильтр по `user_id`
    
- диапазон по `created_at`
    
- сортировка по `created_at DESC`
    
- маленький `LIMIT`
    
- нужны поля `amount`, `status`
    

### Возможный индекс

```sql
CREATE INDEX idx_orders_user_created_desc
ON orders(user_id, created_at DESC)
INCLUDE (amount, status);
```

### Почему так

- `user_id` - точное равенство
    
- `created_at DESC` - диапазон плюс сортировка
    
- `amount`, `status` - покрытие
    

### Что проверить

```sql
EXPLAIN ANALYZE
SELECT id, created_at, amount, status
FROM orders
WHERE user_id = 12345
  AND created_at >= now() - interval '30 days'
ORDER BY created_at DESC
LIMIT 50;
```

### Что ожидаем

- отсутствие full scan
    
- отсутствие отдельной сортировки
    
- маленькое количество чтений
    
- быстрый limit
    

---

## Как понять, что индекс лишний

Признаки:

- почти не используется
    
- дублируется более широким индексом
    
- не ускоряет реальные запросы
    
- заметно замедляет запись
    
- большой по размеру
    
- статистика использования низкая
    

---

## Удаление индекса

```sql
DROP INDEX idx_name;
```

Но удалять надо осторожно и после анализа реальной нагрузки.

## Краткая памятка в конце

### Что запомнить в первую очередь

1. Индекс ускоряет чтение, но замедляет запись.
    
2. Самый важный тип индекса - B-tree.
    
3. Индекс полезен не всегда.
    
4. Важны селективность и кардинальность.
    
5. Порядок столбцов в составном индексе критичен.
    
6. Индекс делают под запрос, а не просто под колонку.
    
7. `LIKE 'abc%'` - нормально для B-tree, `LIKE '%abc%'` - обычно нет.
    
8. Функция над колонкой часто ломает обычный индекс.
    
9. `EXPLAIN ANALYZE` важнее догадок.
    
10. Много индексов - не всегда хорошо.
    
11. FK часто стоит индексировать.
    
12. Partial и functional индексы могут дать огромный выигрыш.
    
13. В PostgreSQL полезно знать GIN, GiST, BRIN, INCLUDE, pg_trgm.
    
14. В MySQL InnoDB важно помнить, что PK кластерный.
    
15. В Oracle важно помнить про bitmap, reverse key и NULL в B-tree.

## Сверхкраткая версия для собеса

**Индекс** - это дополнительная структура данных для быстрого поиска строк.  
**Плюсы** - ускоряет `SELECT`, `JOIN`, `ORDER BY`.  
**Минусы** - занимает место, замедляет `INSERT/UPDATE/DELETE`.  
**Основной тип** - `B-tree`.  
**Кластерный индекс** определяет физический порядок данных, **некластерный** хранит ключи отдельно.  
**Составной индекс** важен по порядку столбцов, работает слева направо.  
**Функциональный индекс** нужен для выражений вроде `LOWER(email)`.  
**Partial index** индексирует только часть строк.  
Индекс может **не использоваться**, если таблица маленькая, условие не селективно или запрос ломает возможность применения индекса.  
Главный инструмент проверки - **EXPLAIN / EXPLAIN ANALYZE**.

## Мини-набор готовых фраз для ответа на собеседовании

**Почему индекс не использовался?**  
Потому что оптимизатор оценил full scan как более дешевый, либо индекс не подходил под условие, либо статистика была неактуальна.

**Как выбрать порядок колонок в составном индексе?**  
Обычно сначала колонки с точным равенством, потом диапазон, потом сортировка, затем дополнительные include-поля.

**Когда индекс вреден?**  
Когда таблица часто изменяется, а индекс почти не участвует в чтении, либо когда индексов слишком много.

**Что бы вы сделали, если запрос медленный?**  
Посмотрел бы `EXPLAIN ANALYZE`, оценил селективность условий, наличие sort, join strategy, статистику, а уже потом решал бы, нужен ли новый индекс, переписывание запроса или партиционирование.
