## Исходный конспект

### Поведенческие шаблоны
Шаблоны, определяющие взаимодействие между объектами и распределение ответственностей.

1.  **Строитель (Builder):** Отделяет конструирование сложного объекта от его представления (например, конвертация документа в разные форматы).
2.  **Наблюдатель (Observer):** Оповещает всех подписчиков об изменении состояния субъекта. Варианты: Push (передача данных) и Pull (наблюдатель сам запрашивает данные).
3.  **Шаблонный метод (Template Method):** Определяет скелет алгоритма в базовом классе, позволяя наследникам переопределять отдельные шаги (хуки).
4.  **Посредник (Mediator):** Централизует общение группы объектов, избавляя их от необходимости знать друг о друге напрямую.
5.  **Команда (Command):** Инкапсулирует запрос в виде объекта. Позволяет ставить команды в очередь, логировать их и реализовывать отмену операций (Undo/Redo).
6.  **Цепочка ответственности (Chain of Responsibility):** Передача запроса по цепочке обработчиков, пока один из них его не обработает (например, всплытие событий в UI или обработка исключений).
7.  **Состояние (State):** Позволяет объекту менять свое поведение при изменении его внутреннего состояния (внешне выглядит как смена класса объекта).
8.  **Посетитель (Visitor):** Позволяет добавлять новые операции к объектам, не меняя их классы. Реализует механизм двойной диспетчеризации.

## Дополнения и пояснения

### Общая идея поведенческих шаблонов
Если структурные паттерны отвечают на вопрос, **как объекты организованы**, то поведенческие — **как они взаимодействуют и как распределяются обязанности**.

### Когда они особенно полезны
Поведенческие шаблоны уместны, когда в системе нужно:
- убрать жёсткую связанность между участниками взаимодействия;
- вынести алгоритм или запрос в отдельный объект;
- централизовать координацию;
- поддержать расширяемость поведения без переписывания существующего кода.

## Примеры

### Observer
Типичный пример — система уведомлений:
- объект `NewsFeed` публикует событие;
- подписчики `EmailSubscriber`, `PushSubscriber`, `SmsSubscriber` получают уведомление.

### Command
Пример — редактор текста:
- команда `InsertText`
- команда `DeleteSelection`
- команда `Undo`

Хранение операций как объектов позволяет строить историю действий.

### State
Пример — TCP-соединение:
- `Closed`
- `Listening`
- `Established`

Поведение при событии зависит от текущего состояния.

## Что полезно запомнить
- Поведенческие паттерны описывают логику взаимодействия.
- Observer полезен для событий.
- Command полезен для очередей и undo/redo.
- State помогает убрать большие `if/switch` по состояниям.
