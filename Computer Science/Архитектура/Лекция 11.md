## Исходный конспект

### Предметно-ориентированное проектирование (DDD)
Методология, фокусирующаяся на анализе предметной области и создании глубокой модели, выраженной в коде.

### Ключевые концепции
1.  **Единый язык (Ubiquitous Language):** Набор терминов, который используют и эксперты, и разработчики, и который напрямую отражен в коде (имена классов и методов).
2.  **Связь модели и кода:** Модель должна эволюционировать вместе с кодом. Если они расходятся — модель бесполезна.
3.  **Изоляция предметной области:** Бизнес-логика должна быть отделена от UI и инфраструктуры (БД).

### Элементы модели
*   **Сущность (Entity):** Объект, обладающий уникальной идентичностью, сохраняющейся во времени (например, Клиент, Груз).
*   **Объект-значение (Value Object):** Объект, определяемый только своими атрибутами. Обычно иммутабелен (например, Адрес, Цвет).
*   **Служба (Service):** Операция, которую нельзя естественным образом отнести к сущности или значению.
*   **Агрегат (Aggregate):** Группа связанных объектов, доступ к которым осуществляется только через один корневой объект (Root). Обеспечивает целостность данных и инвариантов.

### Паттерны инфраструктуры
*   **Репозиторий (Repository):** Иллюзия коллекции объектов в памяти, скрывающая детали работы с БД. Используется только для корней агрегатов.
*   **Фабрика:** Отвечает за создание сложных агрегатов в консистентном состоянии.
*   **Спецификация (Specification):** Инкапсуляция логических условий для проверки объектов или их выборки из репозитория.

## Дополнения и пояснения

### Почему DDD применяют не везде
DDD особенно полезен там, где предметная область действительно сложна:
- много правил;
- много исключений;
- много бизнес-ограничений;
- важна эволюция модели.

Для очень простых CRUD-систем DDD может быть избыточным.

### Единый язык как практический инструмент
Если в коде класс называется `Contract`, в документации пишут `Agreement`, а заказчик говорит `Договор`, команда быстро начинает путаться.  
DDD требует, чтобы ключевые термины были едиными:
- в коде,
- в обсуждениях,
- в документации,
- в тест-кейсах.

## Примеры

### Entity
`Customer` остаётся тем же клиентом, даже если у него изменились адрес и телефон.

### Value Object
`Money(amount, currency)` определяется только значениями.  
Если значения равны, объекты считаются равными по смыслу.

### Aggregate
Агрегат `Order` может включать:
- сам заказ;
- позиции заказа;
- адрес доставки.

Но внешний код должен менять их через корень агрегата `Order`, чтобы сохранить инварианты.

## Что полезно запомнить
- DDD нужен для сложной предметной области.
- Единый язык связывает экспертов и разработчиков.
- Entity, Value Object и Aggregate — базовые строительные блоки модели.
- Репозиторий и фабрика — это инфраструктурные паттерны поддержки модели.
