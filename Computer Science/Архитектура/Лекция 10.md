## Исходный конспект

### Архитектурные стили и шаблоны
Архитектурный стиль накладывает стратегические ограничения на всю систему.

### Шаблоны взаимодействия с пользователем
1.  **Трехзвенная архитектура (3-tier):** Клиент (UI) — Бизнес-логика — Данные (БД). UI не хранит состояние, логика не знает о способе отображения.
2.  **MVC (Model-View-Controller):** Разделение данных (Model), отображения (View) и обработки ввода (Controller).
    *   **Вариации:** MVVM (использует View Model для удобного связывания данных в WPF/Android) и MVP.
3.  **Sense-Compute-Control:** Цикл для робототехники: считать датчики, вычислить воздействие, выдать на актуаторы.

### Структурные стили
1.  **Слоистая архитектура (Layered):** Каждый слой использует только слои ниже. Строгая (только соседний снизу) и нестрогая (любой нижний).
    *   **Пример:** Стандартный 4-уровневый стиль DDD: UI — Приложение — Домен — Инфраструктура.
2.  **Клиент-сервер:** Сервер хранит данные и не знает о количестве клиентов; клиенты общаются только с сервером.
3.  **Гексагональная архитектура (Порты и адаптеры):** Доменная модель в центре, взаимодействие с внешним миром (БД, UI, Веб) идет через адаптеры. Поток управления направлен внутрь.
4.  **Чистая архитектура (Clean Architecture):** Развитие идеи слоев с четким потоком управления через контроллеры, интеракторы и презентеры.

### Стили обработки данных
1.  **Пакетная обработка (Batch):** Последовательное исполнение программ, обменивающихся файлами (стиль Linux pipes).
2.  **Каналы и фильтры (Pipes and Filters):** Потоковая обработка, где фильтры не знают друг о друге и общаются только через каналы. Хорошо параллелится, но плоха для интерактива.
3.  **Доска объявлений (Blackboard):** Общая структура данных, с которой работают независимые агенты (решатели). Пример: оптимизирующие правила в компиляторах.

### Событийные и распределенные стили
1.  **Издатель-подписчик (Pub-Sub):** Отвязывает отправителя от получателя. Бывают шины событий и системы Event Sourcing (состояние строится по логу событий).
2.  **Peer-to-Peer (P2P):** Одноранговые сети (BitTorrent), где каждый узел выполняет все функции и система максимально устойчива к отказам.

## Дополнения и пояснения

### Что такое архитектурный стиль
Архитектурный стиль — это набор ограничений и соглашений, которые определяют:
- как делить систему на части;
- как части взаимодействуют;
- где проходят границы ответственности.

Именно поэтому стиль влияет на систему стратегически, а не локально.

### Почему нельзя выбрать один «лучший» стиль
Один стиль может быть хорош для:
- корпоративной бизнес-системы,

а другой — для:
- IoT,
- data pipeline,
- высоконагруженного web-сервиса,
- desktop-приложения.

Следовательно, выбор зависит от контекста.

## Примеры

### Layered Architecture
Типичный вариант:
- Presentation
- Application
- Domain
- Infrastructure

Преимущество — понятная структура и контролируемые зависимости.

### Hexagonal Architecture
Есть доменная модель в центре.  
Снаружи расположены адаптеры:
- REST API
- база данных
- очередь сообщений
- CLI

Это позволяет менять внешние технологии без переписывания домена.

### Pub-Sub
Пример — сервис заказов публикует событие `OrderCreated`, а сервисы платежей, уведомлений и аналитики получают его независимо друг от друга.

## Что полезно запомнить
- Архитектурный стиль задаёт правила организации системы.
- Layered, Hexagonal и Clean Architecture помогают изолировать бизнес-логику.
- Pub-Sub и Pipes-and-Filters полезны там, где важна слабая связанность и поток данных.
