## Исходный конспект

### Суть моделирования
*   **Определение:** Модель — это упрощенное подобие объекта, позволяющее делать утверждения относительно оригинала.
*   **Цели:** Борьба со сложностью ПО, средство коммуникации между разработчиками и заказчиком, документирование решений.
*   **Принцип:** «Все модели неправильны, но некоторые полезны». Нужно моделировать только то, что существенно для принятия решений.

### Виды моделей
1.  **Естественный язык:** Выразителен для первичного анализа, но не пригоден для автоматической обработки.
2.  **Неформальные графические модели:** Наброски в Powerpoint/Excel для быстрой передачи информации.
3.  **Формальные графические языки (UML):** Имеют строгий синтаксис и иногда семантику.
4.  **Текстовые архитектурные языки (AADL):** Используются для критических систем (авиация), позволяют анализировать надежность и время отклика до написания кода.

### UML (Unified Modeling Language)
UML — это международный стандарт, включающий **14 видов диаграмм**, разделенных на структурные и поведенческие.

#### Диаграмма классов (Class Diagram)
Самая популярная диаграмма, описывающая типы объектов и связи.
*   **Элементы:** Прямоугольники классов (атрибуты и операции).
*   **Отношения:**
    *   **Наследование (Generalization):** Стрелка к родителю.
    *   **Ассоциация:** Линия связи, может иметь кратность (например, `*` — «много»).
    *   **Агрегация (пустой ромб):** Слабая связь.
    *   **Композиция (закрашенный ромб):** Сильное владение (части уничтожаются с целым).
    *   **Зависимость (пунктир):** Общее знание о другом классе.

#### Другие структурные диаграммы
*   **Диаграмма объектов:** Снимок системы в рантайме (конкретные экземпляры и значения полей).
*   **Диаграмма пакетов:** Группировка элементов для организации иерархии модели.
*   **Диаграмма компонентов:** Описывает высокоуровневые модули; наиболее стабильна при изменениях кода.

## Дополнения и пояснения

### Зачем нужна модель, если уже есть код
Код отвечает на вопрос **«как система реализована»**, а модель часто отвечает на вопрос **«как о ней думать»**.  
Поэтому хорошая модель нужна не вместо кода, а как инструмент:
- обсуждения,
- анализа,
- документирования,
- передачи знаний.

### Когда модель особенно полезна
Модель даёт наибольшую ценность, если:
- систему нужно объяснить новой команде;
- проект распределён по нескольким разработчикам;
- важно согласовать понимание с заказчиком;
- требуется зафиксировать архитектурные границы.

### Ограничение UML
UML — это язык описания, но не гарантия хорошего дизайна.  
Даже очень подробная диаграмма не спасёт систему, если:
- границы компонентов выбраны неудачно;
- сущности определены неправильно;
- в модели смешаны разные уровни абстракции.

## Примеры

### Пример диаграммы классов
Для интернет-магазина на диаграмме классов можно выделить:
- `Customer`
- `Order`
- `OrderItem`
- `Product`
- `Payment`

Связи:
- у `Customer` может быть много `Order`;
- `Order` содержит несколько `OrderItem`;
- каждый `OrderItem` связан с одним `Product`.

### Пример полезной диаграммы компонентов
На уровне архитектуры может быть достаточно показать 4 компонента:
- Web UI
- API / Backend
- Database
- External Payment Gateway

Такая диаграмма не описывает внутренности классов, но отлично помогает понять общую структуру решения.

## Что полезно запомнить
- Модель всегда упрощает реальность.
- Хорошая модель — это полезная модель.
- UML особенно ценен как общий визуальный язык команды.
- Не надо рисовать диаграммы ради формальности.
