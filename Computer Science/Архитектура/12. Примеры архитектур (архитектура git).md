## Исходный конспект

### История систем контроля версий (VCS)
От дельта-компрессии в 1975 году (для экономии места) до современных распределенных систем.

### Архитектура Git
Разработан Линусом Торвальдсом в 2005 году для ядра Linux.
*   **Модель данных:** Git — это большая хэш-таблица (Content-addressable storage), где ключом является SHA-1 хэш содержимого.
*   **Объекты (в папке `.git/objects`):**
    *   **Blob:** Содержимое файла (без имени).
    *   **Tree:** Список имен файлов и ссылок на другие деревья или блобы (структура папок).
    *   **Commit:** Ссылка на дерево, автор, сообщение и ссылки на родителей (история).
*   **Ссылки (Refs):** Ветки и теги — это просто текстовые файлы, содержащие хэш коммита.
*   **Механизмы:**
    *   **Index:** Временное хранилище для подготовки коммита.
    *   **Pack-файлы:** Механизм сжатия и упаковки объектов для передачи по сети.
    *   **Reflog:** Лог всех перемещений указателей веток для восстановления данных.

### Архитектура Mercurial (Hg)
Появилась одновременно с Git, написана на Python.
*   **Revlog:** Единый формат хранения записей константного размера с дельта-компрессией.
*   **Changelog и Manifest:** Иерархические структуры для отслеживания релизов и состава файлов в них.
*   **Ветки:** Реализованы как метаданные внутри каждого коммита (именованные ветки).
*   **Расширяемость:** Мощная система плагинов и хуков на Python, работающих в адресном пространстве процесса.

## Дополнения и пояснения

### Почему Git и Mercurial важны архитектурно
Системы контроля версий — это не просто инструменты хранения истории, а важная часть инженерной практики:
- воспроизводимость изменений;
- совместная работа команды;
- безопасные эксперименты;
- аудит решений.

### Главное о Git
Git устроен как content-addressable storage:  
объекты адресуются не именем, а хэшем содержимого.  
Это даёт:
- неизменяемость объектов;
- эффективное переиспользование;
- удобное хранение истории.

## Примеры

### Пример объекта Blob
Если два файла в разных папках имеют одинаковое содержимое, Git может хранить это содержимое как один и тот же blob.

### Пример commit
Коммит связывает:
- снимок дерева файлов;
- автора;
- сообщение;
- родителей.

Поэтому commit — это не “список изменений”, а зафиксированное состояние проекта плюс метаданные.

### Пример reflog
Если ветка случайно была передвинута, reflog может помочь найти прежнее положение указателя и восстановить потерянный коммит.

## Что полезно запомнить
- Git хранит содержимое как хэшируемые объекты.
- Blob, Tree и Commit — три ключевых типа объектов.
- Ветки — это ссылки на коммиты.
- Reflog помогает восстановить потерянную историю.
