
## Вероятностные структуры данных и где они обитают
https://habr.com/ru/articles/743800/
### Что это такое

Вероятностные структуры данных нужны там, где точный ответ либо слишком дорог, либо вообще не нужен. Вместо строгого “да / нет” или точного числа они дают ответ с контролируемой ошибкой: например, “элемента точно нет”, “элемент, возможно, есть”, “уникальных значений примерно столько-то” или “этот объект встречался не реже N раз”. Их главный смысл — сильно экономить память и время на больших потоках данных и массивах событий. ([Wikipedia](https://en.wikipedia.org/wiki/Bloom_filter?utm_source=chatgpt.com "Bloom filter"))

У этих структур обычно есть понятная цена за эффективность. Для фильтра Блума это ложноположительные ответы, для HyperLogLog — оценка с небольшой статистической погрешностью, для Count-Min Sketch — завышение частот из-за коллизий. Зато все три структуры хорошо подходят для потоковой обработки, легко масштабируются и часто поддерживают слияние нескольких экземпляров в один агрегированный результат. ([Wikipedia](https://en.wikipedia.org/wiki/Bloom_filter?utm_source=chatgpt.com "Bloom filter"))

## Фильтр Блума

### Идея
Проверит, принадлежит ли заданный элемент к множеству. Фильтр Блума довольно быстро обрабатывает большие объёмы данных и возвращает ответ в виде «нет»/«возможно». Стоит отметить, что фильтр Блума широко используется при поиске слов в тексте.

Фильтр Блума представляет собой битовый массив из M элементов и k хэш-функций, каждая из которых возвращает значения в диапазоне от 0 до M-1.

![[Pasted image 20260324134714.png]]

Технически фильтр состоит из битового массива длины `m` и `k` хэш-функций. При добавлении элемента мы вычисляем `k` позиций и выставляем соответствующие биты в 1. При проверке снова считаем те же `k` позиций: если хотя бы один бит равен 0, элемента точно нет; если все равны 1, элемент, возможно, есть. Чем больше элементов мы добавили в фиксированный массив, тем выше вероятность ложноположительного ответа. ([Wikipedia](https://en.wikipedia.org/wiki/Bloom_filter?utm_source=chatgpt.com "Bloom filter"))
![[Pasted image 20260324134851.png|700]]

### Почему он полезен

Bloom filter особенно хорош как “дешёвый предварительный барьер”. Он позволяет быстро отсеять подавляющее большинство заведомо неподходящих элементов и не ходить к более дорогой структуре: на диск, в сеть, в большую таблицу или в полный индекс. Поэтому его часто применяют в системах хранения, базах данных, кэширующих слоях и сетевой аналитике. ([PostgreSQL](https://www.postgresql.org/docs/current/bloom.html "PostgreSQL: Documentation: 18: F.6. bloom — bloom filter index access method"))

### Где встречается

В PostgreSQL `bloom` — это не “контейнер для значений”, а специальный метод доступа к индексу, основанный на Bloom filters. Он хранит сигнатуры, умеет быстро исключать заведомо неподходящие кортежи, но остаётся lossy-индексом, поэтому найденные совпадения нужно перепроверять по реальным значениям строк. Такой индекс особенно полезен, когда есть много столбцов и запросы проверяют разные комбинации равенств. ([PostgreSQL](https://www.postgresql.org/docs/current/bloom.html "PostgreSQL: Documentation: 18: F.6. bloom — bloom filter index access method"))

В HBase Bloom filters используются для оптимизации высоконагруженных запросов; в справочнике HBase прямо сказано, что система поддерживает Block Cache и Bloom Filters для ускорения запросов, а row-based Bloom filters включены по умолчанию начиная с HBase 0.96. ([Apache HBase](https://hbase.apache.org/book.html "Apache HBase® Reference Guide"))

### Ограничения

У обычного Bloom filter есть два важных ограничения: он не умеет хранить сами значения, только “следы” от них, и он не умеет безопасно удалять элементы без дополнительных модификаций вроде counting Bloom filter. Поэтому его почти всегда используют не вместо основной структуры, а вместе с ней. ([Wikipedia](https://en.wikipedia.org/wiki/Bloom_filter?utm_source=chatgpt.com "Bloom filter"))

## HyperLogLog

### Идея
HyperLogLog решает другую задачу: не проверку принадлежности, а приблизительный подсчёт числа различных элементов, то есть **cardinality estimation**. Он особенно полезен там, где точный `COUNT(DISTINCT ...)` слишком дорог по памяти и времени. В Redis реализация HyperLogLog использует до 12 КБ памяти и даёт стандартную ошибку около 0.81%; в классической работе Flajolet и соавторов показано, что оценка множеств порядка `10^9` элементов может достигаться примерно с 2% ошибкой при очень малом объёме памяти. ([Redis](https://redis.io/docs/latest/develop/data-types/probabilistic/hyperloglogs/ "HyperLogLog | Docs"))

Интуиция у алгоритма красивая: если хэш-функция распределяет значения равномерно, то редкие двоичные шаблоны вроде длинной последовательности нулей начинают появляться только на больших выборках. Поэтому по статистике “редкости” таких шаблонов можно оценивать мощность множества. На практике HyperLogLog делит поток по регистраторам, в каждом запоминает максимум по числу ведущих нулей, а затем собирает итоговую оценку через гармоническое среднее и корректировки смещения. ([Algorithms](https://algo.inria.fr/flajolet/Publications/FlFuGaMe07.pdf?utm_source=chatgpt.com "HyperLogLog: the analysis of a near-optimal cardinality ..."))

HyperLogLog основан на двух математических фактах:

1. У «хороших» хэш-функций значения распределены равномерно.
    
2. В случайном потоке двоичных чисел фиксированной длины (например, 32 или 64-битных чисел) суффикс из нулей длины k встречается в среднем раз в 2k элементов: поскольку вероятность встретить ноль на каждой позиции равна ½, вероятность встретить k нулей будет равна $(½)^k$.
    
![[Pasted image 20260324135127.png]]

Теперь мы можем построить простой эстиматор: для каждого элемента множества посчитаем хэш и запомним длину максимального нулевого суффикса k. Тогда в нашем множестве будет примерно $2^k$ элементов.

В реальности такая оценка нам не понравится – между $2^{30}$ и $2^{31}$ довольно большая разница, да и закон больших чисел нам говорит, что если ждать достаточно долго, то можно встретить суффикс любого размера.

Для того чтобы сделать более точную оценку, можно взять m независимых хэш-функций, затем для каждого элемента подсчитать длины нулевых суффиксов $R_1$,$R_2$,...,$R_N$ и усреднить результат:

![[Pasted image 20260324135558.png|700]]

Так как считать значения хэш-функций для каждого элемента дорого и элементов в множестве много, то в LogLog (пока что без Hyper) придумали такой трюк: давайте откусим первые M бит у хэша и назовём их номером нашей хэш-функции, а оставшаяся часть будет хэшем. Тогда достаточно посчитать всего 1 хэш, а точность не изменится.

В качестве последнего шага авторы HyperLogLog предлагают использовать гармоническое среднее и добавить множитель для поправки, чтобы увеличить точность:

![[Pasted image 20260324135631.png|700]]
### Где встречается

В Redis HyperLogLog поддерживается отдельными командами `PFADD`, `PFCOUNT` и `PFMERGE`; он предназначен именно для space-efficient оценки числа уникальных элементов и поддерживает слияние нескольких HLL в один. Это делает структуру удобной для аналитики по потокам, журналам событий, уникальным пользователям, IP-адресам, сессиям и просмотрам. ([Redis](https://redis.io/docs/latest/develop/data-types/probabilistic/hyperloglogs/ "HyperLogLog | Docs"))

### Ограничения

HyperLogLog не отвечает на вопрос “какие именно элементы были уникальными” и не годится для последующего перечисления множества. Он хранит только компактную статистическую сводку, из которой можно восстановить только оценку мощности множества, а не сами элементы. ([Redis](https://redis.io/docs/latest/develop/data-types/probabilistic/hyperloglogs/ "HyperLogLog | Docs"))

## Count-Min Sketch

### Идея

Используется для приближенного подсчёта частоты элементов в потоке данных. Эта структура данных представляет поток данных в компактном виде, тем самым позволяя ему быстро отвечать на запросы о частоте встречаемости элементов. Count-Min Sketch задействуется в различных областях: от машинного обучения до обработки естественного языка.

Как и две предыдущие структуры данных, Count-Min Sketch использует несколько $D$ хэш функций, для каждой из которых заводится массив счетчиков длины $W$:
![[Pasted image 20260324140256.png|600]]
Почему берут именно минимум: коллизии могут только увеличивать значения счётчиков, потому что в те же ячейки попадают и другие элементы. Поэтому простая реализация Count-Min Sketch склонна завышать частоты, но не занижать их. В официальной документации Redis прямо указано, что структура использует сублинейную память ценой over-counting из-за коллизий, а очень маленькие оценки часто разумно трактовать как шум. ([Redis](https://redis.io/docs/latest/develop/data-types/probabilistic/count-min-sketch/ "Count-min sketch | Docs"))

Затем для каждого входящего элемента вычисляются $D$ хэшей $h_1$,$h_2$,...,$h_D$, соответствующие позициям в массивах, после чего счётчики в каждом массиве увеличиваются на 1.

![[Pasted image 20260324140331.png|600]]

При запросе частоты встречаемости элемента процесс повторяется – вычисляются $D$ хэшей, а частота вычисляется как min($f_1$, …, $f_D$), чтобы избежать коллизий.

### Где встречается

Count-Min Sketch хорош для heavy hitters и потоковой аналитики: частотность слов, событий, URL, товаров, сетевых пакетов и телеметрии. В Redis эта структура доступна как probabilistic data type с командами инициализации, инкремента, запроса и слияния эскизов. ([Redis](https://redis.io/docs/latest/develop/data-types/probabilistic/count-min-sketch/ "Count-min sketch | Docs"))

### Ограничения

Count-Min Sketch не хранит ключи в явном виде и потому не умеет сам по себе перечислять “все самые популярные элементы”. Он отвечает на вопрос про частоту для уже заданного элемента. Если нужна именно выдача top-k, его часто сочетают с дополнительными структурами. Кроме того, для редких элементов шум коллизий может быть сопоставим с истинным счётом. ([Redis](https://redis.io/docs/latest/develop/data-types/probabilistic/count-min-sketch/ "Count-min sketch | Docs"))

## Как их различать в одной фразе

Если совсем коротко:

- **Bloom filter** — “видели ли мы это раньше?”
    
- **HyperLogLog** — “сколько уникальных объектов было?”
    
- **Count-Min Sketch** — “как часто встречался этот объект?”

# Простейшие реализации на Python

## 1. Bloom Filter

```python
import math
import hashlib


class BloomFilter:
    def __init__(self, capacity: int, false_positive_rate: float = 0.01):
        if capacity <= 0:
            raise ValueError("capacity must be > 0")
        if not (0 < false_positive_rate < 1):
            raise ValueError("false_positive_rate must be between 0 and 1")

        self.capacity = capacity
        self.false_positive_rate = false_positive_rate

        # Оптимальные оценки размера массива и числа хэшей
        self.m = math.ceil(-capacity * math.log(false_positive_rate) / (math.log(2) ** 2))
        self.k = max(1, round((self.m / capacity) * math.log(2)))

        self.bits = bytearray((self.m + 7) // 8)
        self.count = 0

    def _hashes(self, item: str):
        data = item.encode("utf-8")
        for i in range(self.k):
            h = hashlib.blake2b(data, digest_size=8, person=f"h{i}".encode())
            value = int.from_bytes(h.digest(), "big")
            yield value % self.m

    def _set_bit(self, pos: int):
        self.bits[pos // 8] |= 1 << (pos % 8)

    def _get_bit(self, pos: int) -> bool:
        return bool(self.bits[pos // 8] & (1 << (pos % 8)))

    def add(self, item: str):
        for pos in self._hashes(item):
            self._set_bit(pos)
        self.count += 1

    def __contains__(self, item: str) -> bool:
        return all(self._get_bit(pos) for pos in self._hashes(item))

    def info(self):
        return {
            "capacity": self.capacity,
            "false_positive_rate": self.false_positive_rate,
            "bit_array_size": self.m,
            "num_hashes": self.k,
            "inserted_items": self.count,
        }


if __name__ == "__main__":
    bf = BloomFilter(capacity=1000, false_positive_rate=0.01)

    bf.add("apple")
    bf.add("banana")

    print("apple" in bf)   # True
    print("banana" in bf)  # True
    print("orange" in bf)  # False или True, но обычно False
    print(bf.info())
```

### Что здесь важно

- `False` означает “элемента точно нет”.
    
- `True` означает “элемент, возможно, есть”.
    
- Удаления нет.

## 2. HyperLogLog

```python
import math
import hashlib


class HyperLogLog:
    def __init__(self, p: int = 10):
        """
        p - число бит под индекс регистра
        m = 2^p регистров
        """
        if not (4 <= p <= 16):
            raise ValueError("p should be between 4 and 16 for this demo")

        self.p = p
        self.m = 1 << p
        self.registers = [0] * self.m

    def _hash64(self, item: str) -> int:
        data = item.encode("utf-8")
        h = hashlib.blake2b(data, digest_size=8)
        return int.from_bytes(h.digest(), "big")

    @staticmethod
    def _leading_zeros(x: int, bits: int) -> int:
        if x == 0:
            return bits
        return bits - x.bit_length()

    def add(self, item: str):
        x = self._hash64(item)

        # первые p бит -> индекс регистра
        idx = x >> (64 - self.p)

        # оставшиеся биты -> для оценки длины префикса нулей
        w = x & ((1 << (64 - self.p)) - 1)
        rho = self._leading_zeros(w, 64 - self.p) + 1

        self.registers[idx] = max(self.registers[idx], rho)

    def estimate(self) -> float:
        m = self.m

        if m == 16:
            alpha = 0.673
        elif m == 32:
            alpha = 0.697
        elif m == 64:
            alpha = 0.709
        else:
            alpha = 0.7213 / (1 + 1.079 / m)

        z = sum(2.0 ** (-r) for r in self.registers)
        e = alpha * (m ** 2) / z

        # small-range correction (упрощённая)
        v = self.registers.count(0)
        if e <= 2.5 * m and v > 0:
            e = m * math.log(m / v)

        return e

    def merge(self, other: "HyperLogLog"):
        if self.p != other.p:
            raise ValueError("Cannot merge HLL with different precision")
        self.registers = [max(a, b) for a, b in zip(self.registers, other.registers)]


if __name__ == "__main__":
    hll = HyperLogLog(p=10)

    users = ["u1", "u2", "u3", "u1", "u2", "u4", "u5"]
    for user in users:
        hll.add(user)

    print("Оценка числа уникальных:", round(hll.estimate()))
```

### Что здесь важно

- структура не хранит элементы;
    
- она хранит только регистры;
    
- ответ — оценка, а не точное число.

## 3. Count-Min Sketch

```python
import math
import hashlib


class CountMinSketch:
    def __init__(self, width: int, depth: int):
        if width <= 0 or depth <= 0:
            raise ValueError("width and depth must be > 0")

        self.width = width
        self.depth = depth
        self.table = [[0] * width for _ in range(depth)]
        self.total = 0

    @classmethod
    def from_error(cls, epsilon: float, delta: float) -> "CountMinSketch":
        """
        epsilon - допустимая доля ошибки
        delta   - вероятность превысить ошибку
        """
        if not (0 < epsilon < 1 and 0 < delta < 1):
            raise ValueError("epsilon and delta must be between 0 and 1")

        width = math.ceil(math.e / epsilon)
        depth = math.ceil(math.log(1 / delta))
        return cls(width=width, depth=depth)

    def _hashes(self, item: str):
        data = item.encode("utf-8")
        for i in range(self.depth):
            h = hashlib.blake2b(data, digest_size=8, person=f"r{i}".encode())
            value = int.from_bytes(h.digest(), "big")
            yield value % self.width

    def add(self, item: str, count: int = 1):
        if count < 0:
            raise ValueError("This simple version supports only non-negative increments")

        for row, col in enumerate(self._hashes(item)):
            self.table[row][col] += count
        self.total += count

    def estimate(self, item: str) -> int:
        return min(self.table[row][col] for row, col in enumerate(self._hashes(item)))

    def merge(self, other: "CountMinSketch"):
        if self.width != other.width or self.depth != other.depth:
            raise ValueError("Sketch dimensions must match")

        for i in range(self.depth):
            for j in range(self.width):
                self.table[i][j] += other.table[i][j]
        self.total += other.total


if __name__ == "__main__":
    cms = CountMinSketch.from_error(epsilon=0.01, delta=0.001)

    stream = ["apple", "banana", "apple", "orange", "apple", "banana"]
    for item in stream:
        cms.add(item)

    print("apple  ~", cms.estimate("apple"))
    print("banana ~", cms.estimate("banana"))
    print("orange ~", cms.estimate("orange"))
    print("grape  ~", cms.estimate("grape"))
```

### Что здесь важно

- оценка не меньше истинной частоты;
    
- из-за коллизий она может быть больше;
    
- особенно полезен для частых элементов.

## Небольшой итог

> Вероятностные структуры данных полезны там, где объём данных слишком велик для точных, но дорогих вычислений. Bloom filter отвечает на вопрос о возможной принадлежности элемента множеству, HyperLogLog оценивает число уникальных объектов, а Count-Min Sketch приближённо считает частоты в потоке. Все три структуры жертвуют точностью в контролируемых пределах, но взамен дают очень заметную экономию памяти и времени на больших данных.
