# Конспект по пагинации в БД

## 1. Что такое пагинация и зачем она нужна

Пагинация — это способ получать данные **частями**, а не всей таблицей сразу.

Она нужна не только для удобства интерфейса, но и для архитектурной устойчивости системы:

- снижает нагрузку на БД;
- уменьшает объём передаваемых данных;
- ускоряет время ответа;
- защищает сервер и клиент от работы с слишком большими наборами строк;
- делает интерфейс управляемым: страницы, “показать ещё”, бесконечный скролл.

### Где особенно важна пагинация

- списки заказов;
- каталоги товаров;
- новостные ленты;
- история операций;
- логи и аудит;
- административные панели;
- API для мобильных приложений.

---

## 2. Общие требования к хорошей пагинации

Независимо от метода, у хорошей пагинации должны быть:

### 2.1. Детерминированный порядок

Нельзя делать пагинацию без явного `ORDER BY`.

Плохо:

```sql
SELECT * FROM posts LIMIT 20 OFFSET 40;
```

Хорошо:

```sql
SELECT * FROM posts
ORDER BY created_at DESC, id DESC
LIMIT 20 OFFSET 40;
```

Если порядок не зафиксирован, БД не обязана возвращать строки в одном и том же порядке.

---

### 2.2. Устойчивость к одинаковым значениям

Если сортировка идёт по неуникальному полю, например по `created_at`, нужно добавлять второй уникальный ключ, обычно `id`.

Плохо:

```sql
ORDER BY created_at DESC
```

Лучше:

```sql
ORDER BY created_at DESC, id DESC
```

Иначе строки с одинаковым временем могут “прыгать” между страницами.

---

### 2.3. Ограничение размера страницы

Нужно ограничивать максимальный `LIMIT`.

Например:

- по умолчанию `20`;
- максимум `100`.

Иначе клиент может запросить `limit=100000` и создать тяжёлый запрос.

---

## 3. Метод 1: `LIMIT + OFFSET`

## Идея

База пропускает `OFFSET` строк и возвращает следующие `LIMIT`.

### Пример

```sql
SELECT id, title, created_at
FROM posts
ORDER BY created_at DESC, id DESC
LIMIT 20 OFFSET 40;
```

Это значит:

- пропустить первые 40 строк;
- вернуть следующие 20.    

---

## Плюсы

- простой и понятный;
- легко реализуется;
- хорошо ложится на UI с номерами страниц;
- позволяет перейти сразу на страницу 10, 50, 200;
- удобен для фильтров и сортировок в админках.    

---

## Минусы

### 3.1. Деградация производительности

Чем больше `OFFSET`, тем тяжелее запрос.

Причина в том, что БД не “телепортируется” к строке номер 500000. Ей всё равно нужно пройти предыдущие строки, даже если они не попадут в результат.

### 3.2. Нестабильность страниц

Если между запросами кто-то вставил или удалил записи, страницы “съезжают”.

Пример:

- вчера запись была на странице 3;
- сегодня в начало добавили 20 новых строк;
- запись уже на странице 4.

### 3.3. Плохой UX для живых лент

Для ленты событий или чата offset-подход особенно плох, потому что данные меняются постоянно.

---

## Когда `OFFSET` уместен

- маленькие и средние таблицы;
- административные панели;
- отчёты;
- редкие переходы по страницам;
- случаи, где важнее “страница 15”, чем максимальная скорость.

---

## 4. Почему `OFFSET` тормозит даже с индексом

Частая ошибка мышления: “Если есть индекс, значит `OFFSET` будет быстрым”.

Не обязательно.

Индекс помогает находить строки в нужном порядке, но большой `OFFSET` всё равно означает:

- прочитать и отбросить много записей;
- выполнить много шагов сканирования;
- возможно, сделать дополнительные обращения к heap/таблице.

Итог: стоимость всё равно растёт примерно вместе с глубиной страницы.

---

## 5. Метод 2: Keyset / Cursor Pagination

## Идея

Вместо “пропусти 10000 строк” используется принцип:

> “дай следующую страницу после вот этой конкретной записи”

То есть мы передаём не номер страницы, а **якорь** — последнее значение сортировки.

---

## Базовый пример по `id`

### Первая страница

```sql
SELECT id, title
FROM posts
ORDER BY id ASC
LIMIT 20;
```

Допустим, последняя запись на странице имеет `id = 120`.

### Следующая страница

```sql
SELECT id, title
FROM posts
WHERE id > 120
ORDER BY id ASC
LIMIT 20;
```

БД не пропускает 120 записей, а сразу идёт в нужную часть индекса.

---

## Пример с сортировкой по `created_at DESC, id DESC`

### Первая страница

```sql
SELECT id, title, created_at
FROM posts
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

### Следующая страница

Если последняя запись страницы была:

- `created_at = '2024-01-10 12:00:00'`
- `id = 500`

то дальше:

```sql
SELECT id, title, created_at
FROM posts
WHERE (created_at, id) < ('2024-01-10 12:00:00', 500)
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

Это классический keyset-подход.

---

## Плюсы

- очень быстрый на больших объёмах;
- хорошо использует индекс;
- стабилен при вставках и удалениях;
- отлично подходит для “следующая/предыдущая страница” и бесконечного скролла;
- почти всегда лучше для лент и активных таблиц.    

---

## Минусы

### 5.1. Нет прямого перехода на страницу N

Нельзя просто сказать: “дай 500-ю страницу”, если у тебя нет курсора до неё.

### 5.2. Более сложный API

Нужно:

- формировать курсор;
- кодировать его;
- валидировать;
- передавать назад клиенту.

### 5.3. Нужно строго следить за сортировкой

Курсорная пагинация требует аккуратного порядка и корректного индекса.

---

## 6. Что такое курсор на практике

Курсор обычно содержит значения сортировки последней строки страницы.

Например:

```json
{
  "created_at": "2024-01-10T12:00:00",
  "id": 500
}
```

На клиент это обычно не отдают в сыром виде. Его:

- сериализуют;
- кодируют, например в Base64;
- иногда подписывают.

Тогда клиент видит что-то вроде:

```text
eyJjcmVhdGVkX2F0Ijoi...IiwiaWQiOjUwMH0=
```

---

## 7. Что важно для cursor pagination

## 7.1. Сортировка должна быть уникальной

Если сортировка только по `created_at`, а у 100 строк одинаковое время, курсор будет ненадёжным.

Нужен тайбрейкер:

```sql
ORDER BY created_at DESC, id DESC
```

---

## 7.2. Индекс должен соответствовать сортировке

Если запрос такой:

```sql
ORDER BY created_at DESC, id DESC
```

то полезен индекс:

```sql
CREATE INDEX idx_posts_created_id_desc
ON posts (created_at DESC, id DESC);
```

Если порядок в индексе и в запросе расходится, эффективность падает.

---

## 7.3. Нужно уметь ходить назад

Если нужен не только “next”, но и “prev”, обычно делают:

- отдельный курсор назад;
- или зеркальный запрос с обратной сортировкой.

Это чуть сложнее, чем просто `OFFSET`.

---

## 8. Метод 3: Гибридный подход

Это компромисс между:

- удобством `OFFSET`,
- и скоростью keyset pagination.

## Идея

1. Один раз найти “якорь” нужной страницы через `OFFSET`.
2. Дальше листать уже по курсору.

### Когда это полезно

- в интерфейсе нужны номера страниц;
- но листание рядом должно быть быстрым;
- популярные якоря можно кэшировать

---

## Пример сценария

Пользователь открыл страницу 200.

Система:

1. один раз находит запись-якорь около этой позиции;
2. строит курсор;
3. следующие переходы “200 → 201 → 202” идут уже быстро.

---

## 9. Ещё один вариант: seek pagination

Иногда термин **seek pagination** используют как синоним keyset pagination.

Идея та же:

- не смещаться по количеству строк,
- а “искать от последнего значения”.

То есть:

- `OFFSET` = “отсчитай и пропусти”
- `SEEK/KEYSET` = “начни отсюда.

---

## 10. Важная проблема: пагинация и изменения данных

Любая пагинация живёт в реальном мире, где данные меняются.

### При `OFFSET`

Проблемы:

- дубли;
- пропуски;
- смещение страниц.    

### При cursor/keyset

Проблем меньше, но они тоже есть:

- если запись изменила поле сортировки, она может сдвинуться;
- при нестабильной сортировке можно получить пропуск или дубль.    

Поэтому для cursor pagination особенно важно:

- сортировать по почти неизменяемым полям;
- добавлять уникальный tie-breaker.    

---

## 11. Какой метод выбрать

## `LIMIT + OFFSET`

Бери, если:

- таблица не очень большая;
- нужна нумерация страниц;
- это админка или backoffice;
- данные не слишком активно меняются.

## Cursor / Keyset

Бери, если:

- таблица большая;
- есть бесконечный скролл;
- лента часто меняется;
- нужна высокая производительность;
- прямой переход на страницу не нужен.

## Гибрид

Бери, если:

- нужны и номера страниц, и приемлемая скорость;
- есть кэш;
- можно чуть усложнить серверную логику.

---

## 12. Что возвращать в API

Хорошая пагинация — это не только SQL, но и контракт API.

### Для `OFFSET`

Обычно:

```json
{
  "items": [...],
  "page": 3,
  "page_size": 20,
  "total": 1245
}
```

### Для cursor pagination

Обычно:

```json
{
  "items": [...],
  "next_cursor": "....",
  "prev_cursor": "....",
  "has_more": true
}
```

### Важный нюанс про `total`

Для больших таблиц `COUNT(*)` может быть дорогим.

Иногда лучше:

- вообще не отдавать `total`,
- отдавать приблизительное число,
- или считать отдельно и кэшировать.

---

## 13. Частые ошибки при реализации пагинации

### Ошибка 1. Нет `ORDER BY`

Результат становится недетерминированным.

### Ошибка 2. Сортировка по неуникальному полю без `id`

Могут быть дубли и пропуски.

### Ошибка 3. Очень большой `LIMIT`

Можно легко перегрузить БД и сеть.

### Ошибка 4. `OFFSET` на миллионах строк в горячей ленте

Будет медленно и нестабильно.

### Ошибка 5. Неправильный индекс

БД не сможет эффективно выполнять курсорный запрос.

### Ошибка 6. Курсор формируется только из `id`, хотя сортировка по `created_at`

Курсор должен соответствовать `ORDER BY`.

---

## 14. Практические рекомендации по индексам

### Для offset-пагинации

Индекс полезен на поля сортировки:

```sql
CREATE INDEX idx_posts_created_id
ON posts (created_at DESC, id DESC);
```

### Для cursor pagination

Это уже почти обязательно:

- поля `WHERE` и `ORDER BY` должны соответствовать индексу;
- особенно для составного курсора.

---

## 15. Как проверять эффективность

Для PostgreSQL очень полезен:

```sql
EXPLAIN ANALYZE
SELECT ...
```

Смотреть нужно не только на общее время, но и на:

- тип скана;
- количество реально обработанных строк;
- использование индексов;
- стоимость сортировки;
- рост времени при увеличении глубины страницы.

---

## 16. Практическое правило выбора

Можно пользоваться таким простым ориентиром:

### Если нужен UI “страница 1, 2, 3, 4, 5”

Скорее всего:

- `OFFSET`, если объём умеренный;
- гибрид, если объём большой.

### Если нужен endless scroll / feed

Почти всегда:

- cursor/keyset pagination.

### Если это внутренний список на 5–20 тысяч записей

Не надо усложнять — `LIMIT/OFFSET` обычно достаточно.

### Если это таблица на миллионы строк

Сразу думать в сторону keyset.

---

## 17. Короткий вывод

### `LIMIT + OFFSET`

- просто;
- удобно;
- знакомо;
- но плохо масштабируется.

### Cursor / Keyset

- быстро;
- стабильно;
- масштабируемо;
- но сложнее и без прыжка на страницу N.

### Гибрид

- компромисс;
- полезен для сложного UI;
- требует больше инженерной работы.    

---

Это сочетание:
