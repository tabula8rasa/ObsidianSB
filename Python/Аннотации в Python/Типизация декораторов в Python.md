
Папка **«g»** (`g/decorator.py`) посвящена **типизации декораторов** — одной из самых продвинутых тем в системе аннотаций Python.

Главная задача здесь:

> сохранить точную сигнатуру исходной функции после того, как она была обёрнута декоратором

Это важно, потому что без корректной типизации:
- IDE теряет подсказки по аргументам
- анализатор типов перестаёт понимать сигнатуру
- декоратор превращает функцию в что-то вроде `Callable[..., Any]`

---

## 1. В чём проблема типизации декораторов

Обычный декоратор:

- принимает функцию
- возвращает новую функцию-обёртку

Но обёртка обычно объявляется через `*args` и `**kwargs`, и без специальных инструментов типизатор уже не понимает:

- какие именно аргументы принимает исходная функция
- что она возвращает
- как проверять вызовы после декорирования

---

## 2. Что помогает решить проблему

Для типобезопасных декораторов обычно используются два инструмента:

- `ParamSpec` — чтобы сохранить параметры исходной функции
- `TypeVar` — чтобы сохранить тип возвращаемого значения

В новом синтаксисе Python 3.12+ это можно записывать прямо в объявлении функции.

---

## 3. Пример декоратора

```python
import logging
from typing import Callable, ParamSpec, TypeVar

# P = ParamSpec('P')  # python 3.11<=
# T = TypeVar('T')

def add_logging[**P, T](f: Callable[P, T]) -> Callable[P, T]:
    """A type-safe decorator to add logging to a function."""
    def inner(*args: P.args, **kwargs: P.kwargs) -> T:
        logging.info(f'{f.__name__} was called')
        return f(*args, **kwargs)
    return inner
````

---

## 4. Разбор по частям

### Объявление параметров типа

```python
def add_logging[**P, T](...)
```

Здесь:

- `P` — это `ParamSpec`, то есть описание **всех параметров функции**
- `T` — это тип возвращаемого значения

---

### Аргумент декоратора

```python
f: Callable[P, T]
```

Это означает:

> `f` — функция с параметрами `P` и результатом `T`

То есть если `f` принимает `(x: float, y: float)` и возвращает `float`, то это будет сохранено.

---

### Что возвращает декоратор

```python
-> Callable[P, T]
```

Это значит:

> декоратор возвращает новую функцию с **той же сигнатурой**, что и исходная

Именно это и делает декоратор типобезопасным.

---

## 5. Внутренняя обёртка

```python
def inner(*args: P.args, **kwargs: P.kwargs) -> T:
    logging.info(f'{f.__name__} was called')
    return f(*args, **kwargs)
```

### Что важно здесь

#### `*args: P.args`

Означает:

> позиционные аргументы должны быть ровно такими же, как у исходной функции

#### `**kwargs: P.kwargs`

Означает:

> именованные аргументы тоже должны совпадать с исходной сигнатурой

#### `-> T`

Означает:

> обёртка возвращает тот же тип, что и оригинальная функция

---

## 6. Пример использования декоратора

```python
@add_logging
def add_two(x: float, y: float) -> float:
    """Add two numbers together."""
    return x + y
```

После применения декоратора типизатор всё ещё понимает, что:

```python
add_two(x: float, y: float) -> float
```

а не просто `Callable[..., Any]`.

---

## 7. Почему это важно

Если бы декоратор был типизирован плохо, IDE и анализатор потеряли бы информацию о `add_two`.

Например, подсказки по аргументам могли бы исчезнуть.

С `ParamSpec` сигнатура сохраняется, и анализатор понимает, что:

- `add_two(1.0, 2.0)` — корректно
- `add_two("a", "b")` — ошибка типов

---

## 8. Что такое `ParamSpec`

`ParamSpec` — это специальный механизм для захвата **полной сигнатуры функции**.

В отличие от обычного `TypeVar`, который описывает один тип, `ParamSpec` описывает сразу:

- все позиционные параметры
- все именованные параметры

То есть это “тип параметров функции целиком”.

---

## 9. `P.args` и `P.kwargs`

Внутри обёртки `ParamSpec` разбивается на две части:

```python
P.args
P.kwargs
```

Они используются для аннотации:

- `*args`
- `**kwargs`

Это позволяет типизатору точно понять, что обёртка принимает **те же самые аргументы**, что и исходная функция.

---

## 10. Старый синтаксис (Python 3.10–3.11)

До старого синтаксиса до Python 3.12+ нужно было писать так:

```python
from typing import Callable, ParamSpec, TypeVar

P = ParamSpec("P")
T = TypeVar("T")

def add_logging(f: Callable[P, T]) -> Callable[P, T]:
    def inner(*args: P.args, **kwargs: P.kwargs) -> T:
        return f(*args, **kwargs)
    return inner
```

Это полностью эквивалентно, просто более многословно.

---

## 11. Что было до `ParamSpec`

Раньше часто писали так:

```python
from typing import Callable, Any

def decorator(f: Callable[..., Any]) -> Callable[..., Any]:
    ...
```

Проблема этого подхода в том, что теряется вся информация о сигнатуре.

Типизатор уже не знает:

- сколько аргументов должно быть
- какие у них типы
- какой возвращаемый тип

То есть это сильно ослабляет проверку типов.

---

## 12. `Concatenate`

Если декоратор не просто пробрасывает аргументы, а **изменяет сигнатуру**, то одного `ParamSpec` уже недостаточно. В таких случаях используется `Concatenate`.

Он нужен, когда декоратор:

- добавляет новый аргумент
- убирает аргумент
- подставляет объект автоматически

### Идея

Например, декоратор может подставлять `Lock`, `DBSession`, `Request` или другой объект первым аргументом.

Тогда сигнатура уже не “та же самая”, а изменённая — и это надо явно описывать через `Concatenate`.

---

## 13. Практическая польза

Благодаря правильной типизации декораторов:

- сохраняется автодополнение в IDE
- не теряется информация о сигнатуре
- декораторы становятся безопаснее
- mypy может ловить ошибки вызовов после декорирования

Это особенно важно:

- в библиотеках
- во фреймворках
- в middleware
- в системах логирования
- в системах кеширования
- в dependency injection

---

## 14. Главное преимущество

Типизированный декоратор даёт такой эффект:

> функция после декорирования остаётся “для анализатора” такой же понятной, как и до декорирования

То есть разработчик получает все плюсы декоратора, не теряя точность типизации.

---

## 15. Итог

Папка **«g»** учит писать **типобезопасные декораторы**.

Основные идеи:

- декоратор принимает функцию и возвращает функцию
- нужно сохранить сигнатуру исходной функции
- для этого используется `ParamSpec`
- тип результата сохраняется через `TypeVar`
- `P.args` и `P.kwargs` позволяют правильно описать обёртку
- если сигнатура меняется, применяется `Concatenate`

Это одна из самых важных тем для написания качественных библиотек и сложного инфраструктурного кода.
