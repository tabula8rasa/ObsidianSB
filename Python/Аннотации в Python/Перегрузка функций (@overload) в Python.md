
Папка **«h»** (`h/overload.py`) посвящена **перегрузке функций** — механизму, который позволяет описывать разные сигнатуры одной и той же функции для **статического анализатора типов**.

Это особенно полезно, когда функция:

- принимает разные типы аргументов
- возвращает разные типы результата
- и связь между входом и выходом нужно описать **точно**, а не через грубый `Union`

---

## 1. Зачем нужен `@overload`

В Python нельзя написать несколько реальных функций с одним и тем же именем, как в Java или C++:

```python
def process(x: int): ...
def process(x: str): ...
````

Вторая просто перезапишет первую.

Для решения этой проблемы существует декоратор `@overload`, который нужен **не для рантайма**, а для **типизатора** (`mypy`, `pyright`, IDE).

---

## 2. Основная идея

С помощью `@overload` можно сказать анализатору:

- если на входе `None`, функция возвращает `None`
- если на входе `int`, функция возвращает `tuple[int, str]`
- если на входе `bytes`, функция возвращает `str`

То есть мы описываем не просто “набор допустимых типов”, а **точную зависимость результата от аргумента**.

---

## 3. Пример из материалов

```python
from typing import overload


@overload
def process(response: None) -> None:
    ...

@overload
def process(response: int) -> tuple[int, str]:
    ...

@overload
def process(response: bytes) -> str:
    ...


def process(response):
    ...  # actual implementation goes here
```

---

## 4. Как это читать

### Первая перегрузка

```python
@overload
def process(response: None) -> None:
    ...
```

Означает:

> если в `process()` передать `None`, результат тоже будет `None`

---

### Вторая перегрузка

```python
@overload
def process(response: int) -> tuple[int, str]:
    ...
```

Означает:

> если передать `int`, функция вернёт кортеж вида `tuple[int, str]`

---

### Третья перегрузка

```python
@overload
def process(response: bytes) -> str:
    ...
```

Означает:

> если передать `bytes`, функция вернёт `str`

---

### Реальная реализация

```python
def process(response):
    ...
```

Это **единственная настоящая функция**, которая выполняется в рантайме.

---

## 5. Главное правило перегрузки

Шаблон всегда такой:

1. идут несколько объявлений с `@overload`
2. у них нет настоящей реализации, только `...`
3. после них идёт **одна реальная реализация** без `@overload`

---

## 6. Кто что использует

### Статический анализатор

Смотрит **только на overload-сигнатуры**.

### Python в рантайме

Использует **только последнюю реализацию**.

То есть:

- `mypy` и IDE ориентируются на `@overload`
- интерпретатор Python — на обычный `def process(...)`

---

## 7. Почему `@overload` лучше, чем `Union`

Допустим, мы напишем так:

```python
def process(response: int | bytes) -> tuple[int, str] | str:
    ...
```

Формально это верно, но анализатор уже не сможет точно понять:

- если передали `int`, точно ли вернётся `tuple`?
- если передали `bytes`, точно ли вернётся `str`?

Он увидит только общий `Union`.

---

### А `@overload` позволяет выразить точную связь:

- `int -> tuple[int, str]`
- `bytes -> str`
- `None -> None`

Это намного полезнее для:

- автодополнения
- проверки кода
- читаемости API

---

## 8. Пример использования результата

### Случай с `int`

```python
res = process(1)
a = res[0]
```

Поскольку для `int` указано:

```python
process(response: int) -> tuple[int, str]
```

типизатор понимает, что `res` — это кортеж, и выражение:

```python
res[0]
```

корректно.

---

### Случай с `bytes`

```python
new = process(b'')
new.capitalize()
```

Поскольку для `bytes` указано:

```python
process(response: bytes) -> str
```

типизатор понимает, что `new` — это строка, и метод:

```python
new.capitalize()
```

доступен.

---

## 9. Что было бы без `@overload`

Если бы функция была описана через общий `Union`, IDE не могла бы так точно понимать результат.

Например, `new.capitalize()` могло бы не подсветиться как гарантированно корректное, потому что анализатор видел бы что-то вроде:

```python
tuple[int, str] | str | None
```

А у такого объединения метода `.capitalize()` уже нет.

---

## 10. Важный нюанс: перегрузки не выполняются

Функции с `@overload` сами по себе **не работают** как обычные функции.  
Они существуют только как описание интерфейса.

Если оставить только их и не написать реальную реализацию ниже, нормальной работы не будет.

---

## 11. Что должна делать настоящая реализация

Реальная функция обязана уметь обрабатывать **все случаи**, описанные в перегрузках.

Например, логически это могло бы выглядеть так:

```python
def process(response):
    if response is None:
        return None
    if isinstance(response, int):
        return (response, str(response))
    if isinstance(response, bytes):
        return response.decode()
    raise TypeError("Unsupported type")
```

То есть реальная реализация уже сама проверяет тип аргумента и возвращает нужный результат.

---

## 12. Практическая ценность

`@overload` полезен, когда функция:

- по-разному работает с разными типами
- возвращает не просто `Union`, а разные структуры
- должна быть максимально удобной для других разработчиков

Это часто встречается:

- в библиотеках
- в SDK
- в API-обёртках
- в функциях преобразования данных
- в конструкторах и фабриках

---

## 13. Ограничения и особенности

### Перегрузка не создаёт несколько функций

В рантайме функция всё равно одна.

### Перегрузки должны быть согласованы

Реализация должна покрывать все описанные варианты.

### Порядок важен для чтения

Сначала идут все `@overload`, потом реальная реализация.

---

## 14. Инспекция перегрузок

В современных версиях Python перегрузки можно исследовать и в рантайме через `get_overloads()`.

Это уже более редкий сценарий, но такая возможность есть.

---

## 15. Итог

`@overload` нужен для случаев, когда функция имеет **несколько разных режимов работы**, и нужно точно связать:

- тип входного аргумента
- тип возвращаемого значения

Это делает интерфейс функции:

- понятнее
- безопаснее
- удобнее для IDE
- лучше проверяемым анализаторами типов

---

## 16. Главное, что нужно запомнить

- `@overload` работает для **статического анализа**
- реальные вычисления делает **одна обычная функция**
- перегрузка позволяет выразить зависимость вида  
    `int -> tuple`, `bytes -> str`, `None -> None`
- это точнее и полезнее, чем просто `Union`

---
