## Что такое тесты в целом

Тесты — это код, который проверяет, что другой код работает правильно.

Например, у тебя есть функция:

```python
def add(a, b):
    return a + b
```

Тест проверяет, что:

```python
assert add(2, 3) == 5
```

---

## 1. Unit test, или юнит-тест

**Юнит-тест** проверяет **одну маленькую часть программы изолированно**.  
Обычно это одна функция, один метод или один класс.

### Пример

```python
def is_even(x):
    return x % 2 == 0

def test_is_even():
    assert is_even(4) is True
    assert is_even(5) is False
```

### Смысл

Мы проверяем только логику функции `is_even`, не трогая базу данных, сеть, файлы и так далее.

---

## 2. Integration test, или интеграционный тест

**Интеграционный тест** проверяет, как **несколько частей системы работают вместе**.

Например:

- код + база данных
    
- API + сервис авторизации
    
- приложение + очередь сообщений
    

### Пример

Есть функция, которая сохраняет пользователя в БД и потом читает его обратно:

```python
def test_create_user_in_db():
    user_id = create_user("Ivan")
    user = get_user(user_id)
    assert user.name == "Ivan"
```

### Смысл

Тут мы тестируем уже не одну функцию, а взаимодействие компонентов.

---

## 3. End-to-End test, или E2E-тест

**E2E-тест** проверяет **весь сценарий целиком**, почти как настоящий пользователь.

### Пример

Для интернет-магазина:

1. пользователь открывает сайт
    
2. логинится
    
3. кладет товар в корзину
    
4. оплачивает
    
5. видит подтверждение заказа
    

### Смысл

Проверяется весь путь от начала до конца.

---

## 4. Functional test, или функциональный тест

**Функциональный тест** проверяет, что функция системы работает по требованиям бизнеса.

### Пример

Требование:  
“Пользователь не может оформить заказ без адреса доставки”.

Тест:

```python
def test_order_without_address_fails():
    result = create_order(address=None)
    assert result == "Адрес обязателен"
```

### Смысл

Проверяем бизнес-правило, а не внутреннюю реализацию.

---

## 5. Regression test, или регрессионный тест

**Регрессионный тест** нужен, чтобы убедиться, что после изменений **старый рабочий функционал не сломался**.

### Пример

Раньше логин работал. После добавления двухфакторной авторизации нужно проверить, что обычный вход все еще корректен.

### Смысл

Любой старый тест, который защищает от поломок, можно считать частью регрессионного набора.

---

## 6. Smoke test, или smoke-тест

**Smoke test** — это очень быстрая проверка, что система вообще запускается и не разваливается сразу.

### Пример

- приложение стартует
    
- главная страница открывается
    
- API отвечает `200 OK`
    

### Смысл

Это не глубокая проверка, а быстрая: “система вообще жива или нет”.

---

## 7. Acceptance test, или приемочный тест

**Приемочный тест** показывает, что система соответствует ожиданиям заказчика или бизнес-требованиям.

### Пример

Требование:  
“Менеджер должен видеть отчет по продажам за месяц”.

Тест:

- менеджер входит в систему
    
- выбирает месяц
    
- получает отчет
    
- данные отображаются корректно
    

### Смысл

Это тест на уровне “задача действительно решена”.

---

## 8. Performance test, или нагрузочный тест

**Нагрузочный тест** проверяет, как система ведет себя под нагрузкой.

### Пример

- 1000 пользователей одновременно отправляют запросы
    
- измеряется время ответа
    
- проверяется, не падает ли сервис
    

### Смысл

Не про правильность логики, а про скорость и устойчивость.

---

## 9. Mock-тесты

Это не отдельный вид тестов по уровню, а способ тестирования, когда реальные зависимости заменяют заглушками.

### Пример

Функция отправляет email. В тесте мы не хотим реально отправлять письмо:

```python
def test_register_user(mock_email_sender):
    register_user("a@a.com")
    mock_email_sender.send.assert_called_once()
```

### Смысл

Мы проверяем, что код хотел отправить письмо, но не трогаем реальный почтовый сервис.

---

## 10. Генерик тесты

Термин **“генерик тесты”** не такой стандартный, как `unit` или `integration`.

Обычно под этим понимают **переиспользуемые, шаблонные тесты**, которые можно прогонять для разных реализаций.

### Пример

Есть несколько классов, которые должны одинаково работать:

```python
class JsonStorage:
    def save(self, x): ...
    def load(self): ...

class FileStorage:
    def save(self, x): ...
    def load(self): ...
```

Можно написать один общий набор тестов:

```python
def storage_test(storage):
    storage.save("hello")
    assert storage.load() == "hello"
```

И потом запускать его для `JsonStorage`, `FileStorage` и других.

### Смысл

Один и тот же тест применяется к разным реализациям.

---

# Очень коротко по разнице

- **Unit** — проверяем маленький кусок кода отдельно.
    
- **Integration** — проверяем, как части системы работают вместе.
    
- **E2E** — проверяем весь пользовательский сценарий целиком.
    
- **Smoke** — быстрая проверка, что все вообще стартует.
    
- **Regression** — проверка, что старое не сломали.
    
- **Acceptance** — проверка, что требования выполнены.
    
- **Performance** — проверка под нагрузкой.
    
- **Генерик** — общий шаблон тестов для нескольких похожих реализаций.
    

---

## Простой пример на одном проекте

Допустим, есть сайт заказа еды.

**Unit test:**  
проверить, что функция расчета скидки работает правильно.

**Integration test:**  
проверить, что заказ сохраняется в БД и потом читается.

**E2E test:**  
пользователь вошел, выбрал еду, оплатил, получил заказ.

**Smoke test:**  
сайт открылся, логин работает, API отвечает.

**Regression test:**  
после изменения корзины промокоды все еще применяются правильно.

**Acceptance test:**  
заказчик хотел, чтобы клиент видел историю заказов — проверяем, что это действительно есть.

**Performance test:**  
1000 человек одновременно оформляют заказ.
