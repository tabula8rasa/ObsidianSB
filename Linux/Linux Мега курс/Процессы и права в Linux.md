
В системе Linux реализована концепция **«всё есть файл»**, и это касается не только обычных файлов на диске, но и множества объектов ядра: устройств, сокетов, каналов, а также информации о процессах.

Важно понимать, что фраза **«всё есть файл»** не означает, что любой объект реально хранится как обычный файл на диске. Это значит, что для работы с ним часто используется **файловая абстракция**: его можно читать, иногда записывать, смотреть атрибуты, открывать через файловый интерфейс.

---

## 1. Директория `/proc`

Это **виртуальная файловая система** (`procfs`), которую ядро формирует динамически. Она не лежит на диске как обычная файловая система, а создаётся ядром “на лету” и отражает текущее состояние системы.

Через `/proc` можно получать информацию:

- о памяти,
    
- о процессоре,
    
- о параметрах ядра,
    
- о точках монтирования,
    
- о каждом процессе отдельно.
    

### Системные файлы в `/proc`

- `meminfo` — данные об оперативной памяти.  
    Используется, например, утилитой `free`.
- `cpuinfo` — информация о процессоре: модель, частота, количество ядер, флаги, `bogomips`.
    
- `config.gz` — конфигурация текущего ядра, если она включена в сборку ядра.
    
- `cmdline` — параметры, переданные ядру загрузчиком при старте системы.
    
- `filesystems` — список файловых систем, которые поддерживает ядро.
- `mounts` — информация о текущих точках монтирования.

### Полезные дополнительные файлы в `/proc`

- `uptime` — сколько система работает с момента загрузки.
    
- `loadavg` — средняя нагрузка на систему.
    
- `version` — версия ядра.
    
- `swaps` — информация о swap.
    
- `interrupts` — статистика аппаратных прерываний.
    
- `modules` — загруженные модули ядра.

## 2. Директории процессов в `/proc/PID`

Каждому процессу соответствует папка, названная его **PID** — идентификатором процесса.

Например, если процесс имеет PID `1234`, то его каталог будет:

```bash
/proc/1234
```

### Ключевые файлы внутри `/proc/PID`

- `cmdline` — команда, которой был запущен процесс.  
    Аргументы обычно разделены нулевыми байтами, поэтому вывод может выглядеть необычно.
        
    ```bash
    cat /proc/1234/cmdline
    tr '\0' ' ' < /proc/1234/cmdline
    ```
    
- `cwd` — символическая ссылка на текущую рабочую директорию процесса.

    ```bash
    ls -l /proc/1234/cwd
    ```
    
- `exe` — символическая ссылка на исполняемый файл процесса.
    
    ```bash
    ls -l /proc/1234/exe
    ```
    
- `environ` — переменные окружения процесса. Для чтения удобно использовать `strings`.
    
    ```bash
    strings /proc/1234/environ
    ```
    

### Очень полезные дополнительные файлы процесса

- `status` — краткая человекочитаемая информация о процессе: имя, UID, GID, состояние, память.
    
- `stat` — подробная статистика в компактном машинном формате.
    
- `fd/` — открытые файловые дескрипторы процесса.
    
- `maps` — карта памяти процесса.
    
- `limits` — ресурсные лимиты.
    
- `io` — статистика ввода-вывода.
    
- `comm` — короткое имя процесса.
    

Примеры:

```bash
cat /proc/1234/status
ls -l /proc/1234/fd
cat /proc/1234/limits
cat /proc/1234/io
```

### Что полезно знать о PID

- PID — это уникальный номер процесса в рамках текущей системы.
    
- У каждого процесса есть также **PPID** — PID родительского процесса.
    
- Процесс создаётся обычно через `fork()`, а затем может заменить свой код через `exec()`.
    

Посмотреть PID и PPID можно так:

```bash
ps -o pid,ppid,cmd -p 1234
```

## 3. Что такое процесс

**Процесс** — это выполняющаяся программа со своим:

- адресным пространством,
    
- набором открытых файлов,
    
- правами доступа,
    
- переменными окружения,
    
- текущим рабочим каталогом,
    
- контекстом выполнения.
    

Важно отличать:

- **программу** — файл на диске, например `/bin/ls`,
    
- **процесс** — уже запущенный экземпляр этой программы в памяти.
    

Например, один и тот же `python` можно запустить несколько раз — это будут разные процессы с разными PID.

### Основные состояния процесса

Обычно в Linux можно встретить такие состояния:

- `R` — running, выполняется или готов к выполнению,
    
- `S` — sleeping, ожидает событие,
    
- `D` — uninterruptible sleep, обычно ожидание диска или устройства,
    
- `T` — stopped, остановлен,
    
- `Z` — zombie, завершился, но родитель ещё не забрал код завершения.
    

Пример:

```bash
ps aux
ps -eo pid,ppid,state,cmd | less
```

## 4. Мониторинг и управление процессами

Для просмотра процессов используются:

### `ps`

Утилита для получения списка процессов, часто используется в скриптах или для поиска PID.

Примеры:

```bash
ps aux
ps -ef
ps aux | grep nginx
pgrep firefox
pidof bash
```

- `ps aux` — показывает процессы всех пользователей в BSD-формате.
    
- `ps -ef` — похожий вывод в System V-формате.
    

### `top`

Стандартная утилита мониторинга процессов, доступна почти везде. Позволяет видеть загрузку CPU, памяти, список процессов, сортировать их по ресурсам.

Запуск:

```bash
top
```

Полезно внутри `top`:

- `P` — сортировка по CPU,
    
- `M` — сортировка по памяти,
    
- `k` — отправить сигнал процессу,
    
- `r` — изменить nice.
    

### `htop`

Более удобная и визуальная утилита с поддержкой мыши.

Запуск:

```bash
htop
```

Особенности:

- древовидное отображение процессов,
    
- цветной вывод,
    
- удобный поиск,
    
- быстрые действия через клавиши.
    

В `htop` **оранжевая или жёлтая часть памяти** обычно связана с **buffers/cache**, то есть памятью, используемой ядром под кэш и буферы. Это не “потерянная” память — она может быть освобождена при необходимости.

### `atop`

Мощная утилита мониторинга, показывает не только процессы, но и нагрузку на диск, сеть, память, CPU. Поддерживает логирование и анализ истории.

Файл настройки часто находится здесь:

```bash
/etc/default/atop
```

## 5. Завершение и управление процессами

### `kill`

Команда `kill` отправляет процессу сигнал.  
По умолчанию:

```bash
kill PID
```

отправляет **SIGTERM**.

### Часто используемые сигналы

Лучше запоминать **имена**, а не номера, потому что номера могут различаться между архитектурами.

- `SIGTERM` — корректная просьба завершиться.
    
- `SIGKILL` — немедленное принудительное завершение, процесс не может его перехватить.
    
- `SIGINT` — прерывание из терминала, обычно `Ctrl + C`.
    
- `SIGSTOP` — немедленная остановка процесса.
    
- `SIGCONT` — продолжить выполнение остановленного процесса.
    
- `SIGSEGV` — ошибка обращения к памяти, segmentation fault.
    

Примеры:

```bash
kill 1234
kill -TERM 1234
kill -KILL 1234
kill -STOP 1234
kill -CONT 1234
```

### Важное уточнение про `Ctrl + Z`

Терминал обычно отправляет не `SIGSTOP`, а **`SIGTSTP`**, то есть “остановить с терминала”.  
Это важно, потому что:

- `SIGSTOP` нельзя перехватить,
    
- `SIGTSTP` процесс теоретически может обработать.
    

Пример работы с фоновыми задачами:

```bash
sleep 1000
# Ctrl + Z
jobs
bg
fg
```

Где:

- `jobs` — показывает задачи текущей оболочки,
    
- `bg` — продолжить в фоне,
    
- `fg` — вернуть на передний план.
    

### `killall` и `pkill`

Иногда удобнее завершать по имени:

```bash
killall firefox
pkill nginx
pkill -f "python app.py"
```

## 6. Приоритеты процессов: Nice и scheduler

### Nice

Приоритет процесса в пользовательском пространстве задаётся через **nice**:

- от **-20** — самый высокий приоритет,
    
- до **19** — самый низкий,
    
- по умолчанию — `0`.
    

Чем **больше** nice, тем процесс “вежливее” и тем меньше времени CPU он получает по сравнению с другими.

### Запуск с nice

```bash
nice -n 10 command
```

Пример:

```bash
nice -n 10 tar czf backup.tar.gz /home
```

### Изменение nice для уже запущенного процесса

```bash
renice 5 -p 1234
```

Обычно:

- обычный пользователь может только **увеличить** nice своего процесса, то есть понизить его приоритет,
    
- для уменьшения nice, то есть повышения приоритета, обычно нужны права root.
    

### Проверка nice

```bash
ps -o pid,ni,cmd -p 1234
```

## 7. Расширенные права доступа

Стандартные права в Linux:

- `r` — read,
    
- `w` — write,
    
- `x` — execute.
    

Они задаются для:

- владельца файла,
    
- группы,
    
- остальных пользователей.
    

Пример:

```bash
-rwxr-xr--
```

Это означает:

- владелец: `rwx`,
    
- группа: `r-x`,
    
- остальные: `r--`.
    

В числовой форме:

- `r = 4`
    
- `w = 2`
    
- `x = 1`
    

Например:

- `755 = rwxr-xr-x`
    
- `644 = rw-r--r--`

## 8. SUID, SGID, Sticky bit

### SUID (Set User ID)

Позволяет запускать **исполняемый файл** с правами его владельца.

Классический пример — `passwd`, который должен менять системный файл паролей, хотя запускается обычным пользователем.

Пример:

```bash
ls -l /usr/bin/passwd
```

В выводе можно увидеть `s`:

```bash
-rwsr-xr-x
```

Это значит, что на позиции выполнения владельца установлен SUID.

Важно:

- SUID работает в основном на **исполняемых бинарных файлах**.
    
- На скриптах SUID обычно игнорируется по соображениям безопасности.
    

---

### SGID (Set Group ID)

Для файла:

- процесс запускается с правами группы файла.
    

Для директории:

- новые файлы и поддиректории внутри наследуют **группу родительской директории**.
    

Это удобно для общих каталогов проекта.

Пример:

```bash
chmod g+s shared_dir
ls -ld shared_dir
```

Можно увидеть:

```bash
drwxr-sr-x
```

---

### Sticky bit

Если установлен на директорию, то удалять или переименовывать файлы внутри неё может только:

- владелец файла,
    
- владелец директории,
    
- root.
    

Классический пример:

```bash
/tmp
```

Проверка:

```bash
ls -ld /tmp
```

Обычно там будет:

```bash
drwxrwxrwt
```

Буква `t` в конце и есть sticky bit.

## 9. `chmod`, `chown`, `chgrp`

### `chmod`

Меняет права доступа.

Примеры:

```bash
chmod 755 script.sh
chmod u+x script.sh
chmod go-w file.txt
```

### `chown`

Меняет владельца файла.

Пример:

```bash
sudo chown user:filegroup file.txt
```

### `chgrp`

Меняет группу файла.

Пример:

```bash
chgrp developers project.txt
```

## 10. Права по умолчанию и umask

### `umask`

`umask` задаёт маску прав, которая **убирает** разрешения при создании новых файлов и директорий.

Максимальные права:

- для файлов — `666`, потому что обычный файл не получает `x` автоматически,
    
- для директорий — `777`.
    

По умолчанию часто стоит:

```bash
022
```

Тогда:

- новый файл: `666 - 022 = 644`
    
- новая директория: `777 - 022 = 755`
    

Проверка:

```bash
umask
```

Временная установка:

```bash
umask 077
```

Тогда:

- файлы будут обычно `600`,
    
- директории — `700`.
    

Это полезно для приватных данных.

### Важное замечание

`umask` не всегда буквально “вычитается как число”, а применяется как **битовая маска**. Но для практики удобно запоминать именно через модель “отнимается от максимума”.

## 11. ACL — расширенные списки доступа

Стандартной схемы `owner/group/others` иногда недостаточно.  
Тогда используют **ACL (Access Control List)** — расширенные списки прав.

ACL позволяют назначать права:

- конкретному пользователю,
    
- конкретной группе,
    
- сразу нескольким субъектам на один и тот же объект.
    

### Просмотр ACL

```bash
getfacl file.txt
```

### Установка ACL

```bash
setfacl -m u:alice:rwx file.txt
setfacl -m g:devs:rx file.txt
```

### Рекурсивная установка

```bash
setfacl -R -m g:devs:rwx project_dir
```

### ACL по умолчанию для директории

Можно задать **default ACL**, чтобы новые файлы внутри каталога сразу получали нужные права:

```bash
setfacl -d -m g:devs:rwx project_dir
```

### Важное замечание

Не все инструменты резервного копирования по умолчанию сохраняют ACL.  
Например, старые версии `tar` или неудачно выбранные ключи могут потерять эту информацию при восстановлении.

## 12. Атрибуты файлов: `chattr` и `lsattr`

Это дополнительные атрибуты файловой системы, которые работают ниже обычной модели `rwx`.

- `chattr` — установка атрибутов,
    
- `lsattr` — просмотр.
    

Примеры:

```bash
lsattr file.txt
sudo chattr +i file.txt
sudo chattr +a logfile.log
```

### Атрибут `i` — immutable

Файл нельзя:

- изменять,
    
- удалять,
    
- переименовывать,
    
- создавать на его месте новую версию.
    

Пример:

```bash
sudo chattr +i important.conf
```

После этого даже root не сможет просто отредактировать файл обычным образом, пока не снимет атрибут:

```bash
sudo chattr -i important.conf
```

### Атрибут `a` — append only

В файл можно только **добавлять** данные в конец, но нельзя:

- удалить старое содержимое,
    
- перезаписать файл целиком.
    

Пример:

```bash
sudo chattr +a audit.log
echo "new line" >> audit.log
```

Это полезно для логов.

### Важное замечание

Эти атрибуты поддерживаются не всеми файловыми системами одинаково. Чаще всего их используют на `ext2/ext3/ext4`.

## 13. Связь процессов с пользователями и правами

Каждый процесс в Linux имеет:

- **real UID** — реальный пользователь, кто запустил процесс,
    
- **effective UID** — чьи права сейчас используются,
    
- аналогично для GID.
    

Это особенно важно для SUID/SGID-программ.

Посмотреть информацию можно через:

```bash
cat /proc/1234/status
id
```

В `/proc/PID/status` можно увидеть поля:

- `Uid`
    
- `Gid`

## 14. Что ещё полезно знать про процессы

### Файловые дескрипторы

У каждого процесса есть таблица открытых файловых дескрипторов:

- `0` — stdin,
    
- `1` — stdout,
    
- `2` — stderr.
    

Посмотреть можно так:

```bash
ls -l /proc/1234/fd
```

### Лимиты ресурсов

Можно ограничивать:

- число открытых файлов,
    
- размер стека,
    
- число процессов,
    
- объём памяти.
    

Смотреть лимиты:

```bash
ulimit -a
cat /proc/1234/limits
```

### Демоны

Многие системные службы — это фоновые процессы, то есть **демоны**:

- `sshd`
    
- `systemd`
    
- `cron`
    

Посмотреть дерево процессов:

```bash
ps -ef --forest
```

## 15. Практические примеры

### Найти процесс по имени

```bash
ps aux | grep nginx
pgrep nginx
```

### Узнать, откуда запущен процесс

```bash
ls -l /proc/1234/exe
```

### Посмотреть рабочую директорию процесса

```bash
ls -l /proc/1234/cwd
```

### Посмотреть переменные окружения процесса

```bash
strings /proc/1234/environ
```

### Запустить “тяжёлую” задачу с пониженным приоритетом

```bash
nice -n 15 python big_script.py
```

### Поставить процесс на паузу и продолжить

```bash
kill -STOP 1234
kill -CONT 1234
```

### Назначить право конкретному пользователю через ACL

```bash
setfacl -m u:ivan:rw report.txt
getfacl report.txt
```

### Сделать файл неизменяемым

```bash
sudo chattr +i config.cfg
lsattr config.cfg
```

## 16. Короткие важные выводы

1. `/proc` — это интерфейс ядра к текущему состоянию системы и процессов.
    
2. Каждый процесс имеет PID и собственный каталог в `/proc/PID`.
    
3. Для мониторинга процессов чаще всего используют `ps`, `top`, `htop`, `atop`.
    
4. Процессами управляют через сигналы: мягкое завершение — `SIGTERM`, жёсткое — `SIGKILL`.
    
5. `nice` и `renice` позволяют влиять на приоритет выполнения.
    
6. Помимо обычных прав `rwx`, в Linux есть специальные биты: **SUID**, **SGID**, **Sticky bit**.
    
7. `umask` задаёт права по умолчанию для новых файлов и директорий.
    
8. ACL позволяют выдавать права точечно конкретным пользователям и группам.
    
9. `chattr` и `lsattr` работают с атрибутами файловой системы, например `immutable` и `append only`.
    
10. Для понимания Linux очень важно уметь читать `/proc`, смотреть права, понимать сигналы и различать владельца процесса, группу и effective UID.
