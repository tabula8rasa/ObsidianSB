## 1. Протокол Syslog и демон Rsyslog

В современных дистрибутивах Linux системные логи часто обрабатываются демоном **rsyslog** по протоколу **Syslog**.

### Базовая идея

Syslog — это унифицированный механизм передачи и хранения системных сообщений. Разные программы, ядро, сервисы и демоны отправляют сообщения в общую подсистему журналирования, а она уже решает, куда их писать: в файл, в сокет, на удаленный сервер, в базу или в другой обработчик.

### Конфигурация

Основной файл — `/etc/rsyslog.conf`, дополнительные настройки обычно находятся в `/etc/rsyslog.d/`.

Правила задают:

- источник сообщения, то есть **facility**
    
- уровень важности, то есть **priority**
    
- действие, например запись в файл, пересылка на удаленный сервер, вывод в консоль
    

Пример логики правила:

```conf
auth.info    /var/log/auth.log
*.warning    /var/log/warn.log
mail.*       -/var/log/mail.log
```

### Источники сообщений, Facility

Наиболее употребимые:

- `auth`, `authpriv` — аутентификация и авторизация
    
- `cron` — задания планировщика
    
- `kern` — сообщения ядра
    
- `mail` — почтовые службы
    
- `daemon` — обычные системные демоны
    
- `syslog` — внутренние сообщения самого syslog
    
- `user` — пользовательские процессы
    
- `local0`–`local7` — пользовательские категории для своих приложений
    

### Приоритеты

Уровни от наиболее критичного к наименее:

- `emerg`
    
- `alert`
    
- `crit`
    
- `err`
    
- `warning`
    
- `notice`
    
- `info`
    
- `debug`
    

Чем меньше номер, тем критичнее сообщение.

### Особенности записи

Символ `-` перед путем к файлу:

```conf
-/var/log/mail.log
```

означает буферизованную запись. Это ускоряет работу, но при резком выключении питания часть сообщений может не успеть сохраниться.

### Полезные особенности фильтрации

В rsyslog можно использовать:

- маски по facility и priority
    
- исключения
    
- шаблоны формата записи
    
- пересылку на удаленные узлы по TCP и UDP
    
- правила маршрутизации для разных приложений
    

Пример:

```conf
*.info;mail.none;authpriv.none    /var/log/messages
```

Это значит: писать все сообщения уровня info и выше, кроме почтовых и авторизационных.

### Проверка конфигурации

После изменения конфигов важно:

```bash
rsyslogd -N1
systemctl restart rsyslog
systemctl status rsyslog
```

### Утилита logger

Для тестовой отправки сообщения в syslog удобно использовать:

```bash
logger "test message"
logger -p local0.info "application started"
logger -t myapp "custom tag message"
```

Это полезно для проверки, как срабатывают правила маршрутизации логов.

## 2. Ротация логов: Logrotate

Утилита **logrotate** нужна, чтобы логи не разрастались бесконтрольно и не заполняли диск.

Обычно запускается автоматически по расписанию через `cron` или таймеры systemd.

### Основные параметры

- `rotate [число]` — сколько архивов хранить
    
- `create` — создать новый пустой файл после ротации
    
- `compress` — сжимать старые логи
    
- `delaycompress` — не сжимать самый свежий архив сразу
    
- `size` — ротировать по размеру
    
- `daily`, `weekly`, `monthly` — ротировать по времени
    
- `missingok` — не считать ошибкой отсутствие файла
    
- `notifempty` — не ротировать пустой лог
    
- `copytruncate` — копировать и обнулить файл без остановки процесса
    
- `postrotate ... endscript` — команды после ротации
    
- `prerotate ... endscript` — команды до ротации
    
- `sharedscripts` — выполнять скрипт один раз на группу логов
    

### Важное замечание про copytruncate

Параметр `copytruncate` нужен, если процесс не умеет корректно переоткрывать лог после ротации. Но у него есть минус: в момент копирования можно потерять часть строк, если приложение активно пишет в файл.

### Пример конфигурации

```conf
/var/log/myapp/*.log {
    daily
    rotate 7
    compress
    delaycompress
    missingok
    notifempty
    create 0640 root adm
    postrotate
        systemctl reload myapp
    endscript
}
```

### Ручной запуск и отладка

```bash
logrotate -d /etc/logrotate.conf
logrotate -f /etc/logrotate.conf
```

- `-d` — режим отладки, без реального выполнения
    
- `-f` — принудительная ротация
    

### Где лежат конфиги

- основной файл: `/etc/logrotate.conf`
    
- отдельные правила: `/etc/logrotate.d/`

## 3. Systemd Journal, Journald

Современная система логирования в Linux — **systemd-journald**. Она либо дополняет syslog, либо частично заменяет его.

### Главное отличие

Логи хранятся в бинарном формате журнала, а не обязательно как обычные текстовые файлы. Это дает:

- быстрый поиск
    
- фильтрацию по полям
    
- удобную привязку к unit systemd
    
- просмотр логов по загрузкам системы
    

### Хранение

Если используется `Storage=auto`, поведение зависит от наличия `/var/log/journal`.

Если каталога нет, логи могут храниться только до перезагрузки, то есть в volatile-режиме.  
Чтобы журнал сохранялся между перезагрузками, обычно:

```bash
mkdir -p /var/log/journal
systemctl restart systemd-journald
```

или в конфиге указывают:

```ini
[Journal]
Storage=persistent
```

### Основные команды journalctl

```bash
journalctl
journalctl -b
journalctl -b -1
journalctl --list-boots
journalctl -u nginx
journalctl -f
journalctl -xe
journalctl --since "10 min ago"
journalctl --since "2026-03-01 10:00:00" --until "2026-03-01 12:00:00"
journalctl _PID=1234
journalctl _UID=1000
journalctl /usr/sbin/sshd
```

### Что полезно знать

- `-b` — текущая загрузка
    
- `-b -1` — предыдущая загрузка
    
- `-u` — конкретный сервис
    
- `-f` — просмотр в реальном времени
    
- `-xe` — расширенный вывод ошибок и последних событий
    
- `-p err` — только ошибки
    
- `-n 100` — последние 100 записей
    
- `-o short`, `-o json`, `-o verbose` — форматы вывода
    

Пример:

```bash
journalctl -u docker -p warning -n 50
```

### Поля, по которым можно фильтровать

- `_PID`
    
- `_UID`
    
- `_GID`
    
- `_COMM`
    
- `_EXE`
    
- `_SYSTEMD_UNIT`
    
- `_HOSTNAME`
    
- `PRIORITY`
    

### Очистка и ограничения

```bash
journalctl --vacuum-size=1G
journalctl --vacuum-time=1years
journalctl --disk-usage
```

Также можно настроить лимиты в `/etc/systemd/journald.conf`:

- `SystemMaxUse`
    
- `SystemKeepFree`
    
- `RuntimeMaxUse`
    
- `MaxRetentionSec`
    

### Передача логов из приложений

Приложения могут писать в журнал:

- через stdout и stderr, если запущены как systemd unit
    
- через syslog
    
- через специальные библиотеки
    
- через `systemd-cat`
    

Пример:

```bash
echo "hello" | systemd-cat -t myscript
```

## 4. Централизованный сбор логов

На одном сервере смотреть логи удобно, но в реальной инфраструктуре этого мало. Нужен централизованный сбор.

### Зачем это нужно

- единый поиск по множеству серверов
    
- хранение логов даже после падения хоста
    
- корреляция событий между системами
    
- безопасность и аудит
    
- построение дашбордов и алертов
    

### Варианты

#### Rsyslog как клиент и сервер

Можно пересылать сообщения на отдельный лог-сервер.

Пример отправки на удаленный сервер:

```conf
*.* @@logserver.example.com:514
```

- `@` — UDP
    
- `@@` — TCP
    

#### Стек ELK или альтернативы

- **Elasticsearch** — хранение и поиск
    
- **Logstash** — обработка и преобразование
    
- **Filebeat** — легковесный агент для отправки логов
    
- **Fluentd** или **Fluent Bit** — сбор и маршрутизация логов
    
- **Kibana** — визуализация
    

### Практические моменты

При централизованном сборе надо учитывать:

- время на всех серверах должно быть синхронизировано, обычно через NTP
    
- формат логов желательно нормализовать
    
- чувствительные данные надо маскировать
    
- нельзя хранить логи бесконечно без политики retention
    
- нужно разделять системные, прикладные, audit и security логи

## 5. Локальный мониторинг: Atop

**Atop** — мощная консольная утилита мониторинга, которая показывает систему целиком и умеет сохранять историю.

### Основные показатели

#### PRC

- число процессов
    
- число потоков
    
- активные, спящие, зомби
    

#### CPU

- user, system, idle
    
- `wait` — ожидание I/O
    
- `steal` — время, отнятое гипервизором
    
- высокая загрузка CPU не всегда означает проблему, важно смотреть в сочетании с run queue и wait
    

#### CPL, Load Average

Средняя нагрузка за 1, 5 и 15 минут.  
Если значения стабильно выше числа доступных CPU, это повод проверить:

- не упирается ли система в процессор
    
- нет ли очереди на диск
    
- нет ли зависших процессов
    

#### MEM

- `total`
    
- `free`
    
- `cache`
    
- `buffers`
    
- `slab`
    
- `dirty`
    

Низкий `free` сам по себе не всегда плохой признак, потому что Linux активно использует память под кэш.

#### SWP

Параметры swap.  
Если swap активно используется и одновременно растет I/O wait, возможна нехватка RAM.

#### DSK

- чтение и запись
    
- очередь
    
- утилизация
    
- `avio` — среднее время доступа
    

Большое `avio` часто указывает на медленное хранилище или перегруженный диск.

### Параметры процессов

- `VSIZE` — виртуальная память
    
- `RSIZE` — резидентная память
    
- `CPU` — доля CPU
    
- `RDDSK`, `WRDSK` — диск
    
- `ST` — состояние
    
- `EXC` — код завершения
    

### Полезные клавиши

- `Shift+C` — сортировка по CPU
    
- `Shift+M` — по памяти
    
- `Shift+D` — по диску
    
- `g` — перейти к определенному моменту истории
    
- `t` — детализация по потокам
    

### Исторический режим

Atop умеет писать статистику в файл и потом воспроизводить ее:

```bash
atop -r /var/log/atop/atop_20260328
```

Это очень полезно для разбора проблемы задним числом.

## 6. Другие утилиты мониторинга

Помимо atop существуют и другие полезные инструменты.

### iotop

Показывает дисковую активность процессов в реальном времени.

Пример:

```bash
iotop
iotop -o
```

`-o` — показывать только процессы, реально выполняющие I/O.

### iftop

Показывает сетевой трафик между узлами в реальном времени.

Полезно для ответа на вопросы:

- кто загружает канал
    
- в какую сторону идет трафик
    
- какие IP сейчас активны
    

### top и htop

- `top` — базовый встроенный монитор
    
- `htop` — более удобный интерактивный интерфейс
    

Полезно для быстрой диагностики:

- кто ест CPU
    
- кто занимает память
    
- сколько процессов запущено
    
- есть ли зависшие задачи
    

### vmstat

Компактная сводка по CPU, памяти, swap и I/O:

```bash
vmstat 1
```

Особенно полезны поля:

- `r` — очередь на CPU
    
- `si`, `so` — swap in, swap out
    
- `wa` — I/O wait
    

### iostat

Диагностика дисковой подсистемы:

```bash
iostat -xz 1
```

Полезно смотреть:

- `%util`
    
- `await`
    
- `r/s`, `w/s`
    

### free

```bash
free -h
```

Быстрый обзор памяти и swap.

### sar

Позволяет смотреть исторические показатели системы:

```bash
sar -u 1 5
sar -r
sar -n DEV
```

### mpstat

Показывает загрузку по каждому CPU:

```bash
mpstat -P ALL 1
```

### ss и netstat

Для мониторинга сетевых сокетов:

```bash
ss -tulpen
ss -s
```

### lsof

Для ответа на вопросы:

- какой процесс держит файл
    
- кто использует порт
    
- какие файлы открыты процессом
    

Примеры:

```bash
lsof /var/log/syslog
lsof -i :80
```

### dmesg

Показывает сообщения ядра, особенно полезен для:

- ошибок драйверов
    
- проблем дисков
    
- ошибок USB
    
- OOM killer
    
- сбоев файловых систем
    

Пример:

```bash
dmesg -T | tail -n 50
```

## 7. Системы внешнего мониторинга

Для больших инфраструктур локальных утилит недостаточно.

### Zabbix

Цельная система мониторинга:

- сервер
    
- агенты
    
- шаблоны
    
- триггеры
    
- карты
    
- графики
    
- уведомления
    

Можно мониторить:

- CPU
    
- RAM
    
- диск
    
- сеть
    
- процессы
    
- доступность сервисов
    
- содержимое логов
    
- пользовательские метрики
    

### Prometheus + Grafana

Модульный подход.

#### Prometheus

Собирает метрики по HTTP с экспортеров.

Обычно используются:

- `node_exporter` — метрики ОС
    
- `mysqld_exporter` — MySQL
    
- `postgres_exporter` — PostgreSQL
    
- `blackbox_exporter` — проверка доступности по HTTP, TCP, ICMP
    

#### Grafana

Нужна для красивых дашбордов, графиков и панелей.

#### Alertmanager

Рассылает уведомления:

- Telegram
    
- email
    
- Slack
    
- webhook
    

### Важное отличие логов и метрик

- **Логи** отвечают на вопрос: что именно произошло
    
- **Метрики** отвечают на вопрос: насколько сильно изменилась система
    
- **Трейсы** отвечают на вопрос: где именно прошел запрос и сколько занял каждый этап
    

В серьезной эксплуатации обычно используют все три подхода вместе.

## 8. Разница между логированием и мониторингом

Это близкие, но не одинаковые понятия.

### Логирование

Фиксирует события:

- запуск сервиса
    
- ошибку аутентификации
    
- падение процесса
    
- отказ доступа
    
- перезапуск демона
    

Лог — это запись факта.

### Мониторинг

Наблюдает за состоянием системы и реагирует на отклонения:

- CPU выше 90 процентов
    
- свободного места меньше 10 процентов
    
- сервис недоступен
    
- время ответа API выросло
    

Мониторинг — это система наблюдения и оповещения.

### Связь между ними

Часто мониторинг обнаруживает проблему, а логи помогают понять причину.

Пример:

- мониторинг показал рост времени ответа сайта
    
- логи nginx и приложения показали ошибки подключения к БД
    
- dmesg выявил ошибки диска
    
- iostat подтвердил высокую задержку I/O

## 9. Типовые места хранения логов в Linux

Хотя systemd-journald хранит журнал в бинарном виде, многие привычные логи по-прежнему могут быть доступны в `/var/log`.

Часто встречаются:

- `/var/log/syslog`
    
- `/var/log/messages`
    
- `/var/log/auth.log`
    
- `/var/log/secure`
    
- `/var/log/kern.log`
    
- `/var/log/cron`
    
- `/var/log/maillog`
    
- `/var/log/dmesg`
    
- `/var/log/boot.log`
    

Конкретный набор зависит от дистрибутива.

Примеры:

- Debian, Ubuntu часто используют `syslog`, `auth.log`
    
- RHEL, CentOS, Rocky, AlmaLinux чаще используют `messages`, `secure`, `maillog`

## 10. Практика чтения логов

Для анализа логов полезны обычные текстовые инструменты.

### tail

```bash
tail -f /var/log/syslog
tail -n 100 /var/log/auth.log
```

### less

```bash
less /var/log/messages
```

Полезно для удобной навигации.

### grep

```bash
grep -i error /var/log/syslog
grep sshd /var/log/auth.log
```

### awk и sed

Удобны для фильтрации, выделения колонок и быстрой обработки.

Пример:

```bash
grep "Failed password" /var/log/auth.log | awk '{print $(NF-3)}' | sort | uniq -c | sort -nr
```

Это помогает быстро увидеть IP, с которых чаще всего идут неудачные попытки входа.

## 11. Мониторинг ресурсов системы

При анализе состояния Linux обычно смотрят на четыре базовые подсистемы:

- CPU
    
- память
    
- диск
    
- сеть
    

### CPU

Смотреть:

- загрузку ядер
    
- load average
    
- очередь runnable-процессов
    
- user/system/iowait/steal
    

### Память

Смотреть:

- total, used, available
    
- swap
    
- slab
    
- cache
    
- OOM killer в `dmesg` или `journalctl`
    

### Диск

Смотреть:

- место на файловой системе
    
- inode
    
- задержки операций
    
- очередь
    
- процент утилизации устройства
    

Команды:

```bash
df -h
df -i
iostat -xz 1
```

### Сеть

Смотреть:

- пропускную способность
    
- ошибки интерфейса
    
- потери пакетов
    
- количество соединений
    
- состояние сокетов
    

Команды:

```bash
ip -s link
ss -s
iftop
```

## 12. Проверка сервисов и процессов

Мониторинг Linux почти всегда связан с управлением сервисами.

### systemctl

```bash
systemctl status nginx
systemctl is-active nginx
systemctl is-enabled nginx
systemctl restart nginx
```

### Просмотр unit-файлов

```bash
systemctl cat nginx
```

### Просмотр зависимостей и ошибок запуска

```bash
journalctl -u nginx -b
```

### Поиск процесса

```bash
ps aux | grep nginx
pgrep -a nginx
pidof nginx
```

## 13. Мониторинг доступности

Ресурс может быть жив как процесс, но недоступен как сервис. Поэтому важно проверять именно доступность.

Примеры:

- сайт отвечает по HTTP
    
- база принимает соединение
    
- SSH слушает порт
    
- DNS возвращает запись
    

Команды:

```bash
ping host
curl -I http://host
nc -zv host 22
ss -tulpen
```

Внешние системы мониторинга часто выполняют такие проверки автоматически.

## 14. Мониторинг безопасности и аудита

Логирование важно не только для эксплуатации, но и для безопасности.

### Что обычно смотрят

- попытки входа
    
- sudo
    
- смену прав
    
- создание пользователей
    
- ошибки ssh
    
- изменение конфигурации сервисов
    
- подозрительные процессы
    
- массовые ошибки доступа
    

### Полезные файлы и журналы

- `/var/log/auth.log`
    
- `/var/log/secure`
    
- `journalctl -u sshd`
    
- `last`
    
- `lastb`
    
- `who`
    
- `w`

## 15. Типовые причины проблем, которые видно через мониторинг

### Высокий load average

Возможные причины:

- нехватка CPU
    
- ожидание диска
    
- зависшие процессы
    
- слишком много runnable-задач
    

### Высокий iowait

Обычно указывает на:

- медленный диск
    
- перегруженную подсистему хранения
    
- активный swap
    
- долгие операции записи
    

### Рост памяти

Возможные причины:

- утечка памяти
    
- рост page cache
    
- слишком много процессов
    
- неправильно настроенный Java, Python или другой runtime
    

### Заполнение диска

Причины:

- не работает logrotate
    
- приложение слишком много пишет в лог
    
- временные файлы не очищаются
    
- накопились дампы и архивы
    

### Падение сервиса

Что смотреть:

- `systemctl status`
    
- `journalctl -u`
    
- `dmesg`
    
- конфиг сервиса
    
- наличие нужных портов и файлов

## 16. Базовый порядок диагностики проблемы

Когда на Linux-сервере что-то пошло не так, удобно идти по простому алгоритму.

### Шаг 1

Понять симптом:

- сервис не запускается
    
- сервер тормозит
    
- закончился диск
    
- нет сети
    
- пользователи не могут войти
    

### Шаг 2

Проверить общее состояние:

```bash
uptime
top
free -h
df -h
systemctl --failed
```

### Шаг 3

Посмотреть логи:

```bash
journalctl -xe
journalctl -u <service>
dmesg -T | tail
```

### Шаг 4

Проверить узкое место:

- CPU — `top`, `mpstat`
    
- память — `free`, `vmstat`
    
- диск — `iostat`, `iotop`
    
- сеть — `ss`, `iftop`
    

### Шаг 5

Проверить недавние изменения:

- обновления
    
- конфиги
    
- новые релизы
    
- изменение прав
    
- переполнение диска
    
- перезапуски системы

## 17. Различие между текстовыми логами и бинарным журналом

### Текстовые логи

Плюсы:

- удобно читать обычными утилитами
    
- легко копировать и отправлять
    
- просто парсить
    

Минусы:

- медленнее поиск по большим объемам
    
- меньше структурированности
    
- сложнее хранить дополнительные поля
    

### Бинарный journal

Плюсы:

- структурированные записи
    
- быстрый поиск
    
- фильтрация по полям
    
- удобная связь с unit systemd
    

Минусы:

- нужен `journalctl`
    
- напрямую обычным `cat` не прочитать
    
- при повреждении журнала требуется отдельная проверка

## 19. Практические советы по эксплуатации

1. Не полагаться только на один источник данных.  
    Лучше совмещать метрики, логи и состояние сервисов.
    
2. Всегда проверять диск и inode отдельно.  
    Иногда место еще есть, но inode закончились.
    
3. Следить за ротацией логов.  
    Очень частая причина аварий — переполнение `/var/log`.
    
4. Проверять, что критические логи переживают перезагрузку.  
    Иначе после сбоя нужная информация исчезнет.
    
5. Настраивать алерты не только по факту падения, но и по деградации.  
    Например, рост latency, ошибки 5xx, увеличение retry.
    
6. Не хранить чувствительные данные в открытом виде в логах.  
    Пароли, токены, ключи, персональные данные должны маскироваться.
    
7. Для крупных систем лучше использовать централизованный сбор.  
    Иначе при инциденте придется вручную собирать логи с каждого узла.

## 21. Итог

Мониторинг и логирование в Linux — это не один инструмент, а целый набор взаимосвязанных механизмов.

- **Rsyslog** и **Syslog** решают задачу приема, маршрутизации и хранения сообщений.
    
- **Logrotate** не дает логам переполнить диск.
    
- **Journald** обеспечивает современное системное журналирование с удобным поиском.
    
- **Atop**, `top`, `iotop`, `iftop`, `vmstat`, `iostat` и другие локальные утилиты помогают быстро понять текущее состояние системы.
    
- **Zabbix** и **Prometheus + Grafana** позволяют строить внешний мониторинг, графики и уведомления.
    
- Логи помогают понять причину, а мониторинг — вовремя заметить проблему.
