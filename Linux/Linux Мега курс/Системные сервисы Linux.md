
## 1. Системы инициализации: System V Init vs Systemd

Исторически в Linux существовало два основных подхода к управлению загрузкой системы и службами.

### System V Init (классический стандарт)

- Появился в 1983 году в UNIX System V.
    
- Управляет сервисами с помощью **bash-скриптов** (init-скриптов), расположенных в `/etc/init.d/`.
    
- Основан на концепции **runlevel** — уровней загрузки системы.
    
- Каждый сервис запускался и останавливался через отдельный shell-скрипт.
    

#### Плюсы

- простота;
    
- легковесность;
    
- надежность;
    
- минимум компонентов, которые могут сломаться;
    
- полный контроль через обычные shell-скрипты.
    

#### Минусы

- последовательный запуск сервисов, из-за чего загрузка системы медленнее;
    
- слабая информация о зависимостях между службами;
    
- ограниченный встроенный функционал;
    
- сложнее централизованно отслеживать логи и состояние сервисов.
    

### Systemd (современный стандарт)

- Стал стандартом де-факто в большинстве дистрибутивов Linux.
    
- Управляет системой через **юниты** и зависимости между ними.
    
- Поддерживает **распараллеливание запуска**, что ускоряет загрузку.
    
- Включает много встроенных подсистем: сервисы, таймеры, логирование, сеть, монтирование и другое.
    

#### Плюсы

- высокая скорость запуска благодаря распараллеливанию;
    
- централизованное управление службами;
    
- встроенная система логирования **journald**;
    
- удобные команды через `systemctl`;
    
- явная модель зависимостей;
    
- автоматический перезапуск сервисов;
    
- единообразный способ управления системой.
    

#### Критика

- нарушает классическую философию UNIX «делай что-то одно и делай это хорошо»;
    
- является большим и сложным комплексом;
    
- из-за объёма функционала труднее полностью понять все компоненты;
    
- сбои в важной части Systemd могут сильнее затронуть систему, чем в старых, более разрозненных решениях.
    

### Что делает система инициализации

Система инициализации:

- запускается одной из первых после ядра;
    
- поднимает сервисы и окружение;
    
- переводит систему в рабочее состояние;
    
- следит за некоторыми фоновыми службами;
    
- управляет завершением работы и перезагрузкой.
    

### PID 1

Очень важная идея: процесс инициализации в Linux обычно имеет **PID 1**.  
В системах с Systemd это обычно именно `systemd`.

Проверить можно так:

```bash
ps -p 1 -o pid,comm,args
```

## 2. Структура и типы юнитов Systemd

Вместо скриптов Systemd использует файлы конфигурации — **юниты**. По структуре они действительно похожи на `.ini`-файлы.

### Основные секции юнита

#### 1. `[Unit]`

Секция с общей информацией о юните:

- `Description` — описание;
    
- `Documentation` — ссылка на документацию;
    
- `After` — после каких юнитов запускать;
    
- `Before` — до каких юнитов запускать;
    
- `Requires` — жесткая зависимость;
    
- `Wants` — мягкая зависимость.
    

#### 2. Секция типа, например `[Service]`

Содержит параметры, характерные именно для этого типа юнита:

- команда запуска;
    
- тип процесса;
    
- правила перезапуска;
    
- пользователь;
    
- рабочая директория и др.
    

#### 3. `[Install]`

Используется для автозагрузки юнита:

- `WantedBy` — указывает, к какому target юнит будет привязан;
    
- `RequiredBy` — более жесткая форма привязки.
    

### Пример простого service-юнита

```ini
[Unit]
Description=My demo service
After=network.target

[Service]
Type=simple
ExecStart=/usr/local/bin/my-script.sh
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

### Как понимать зависимости

Нужно различать:

- **порядок запуска** — `After=` и `Before=`;
    
- **зависимость по необходимости** — `Requires=` и `Wants=`.
    

Например:

- `After=network.target` означает только порядок;
    
- но не означает, что сеть обязана быть поднята;
    
- если нужна именно обязательная зависимость, обычно используют и `Requires=`, и `After=` вместе.

## 3. Типы юнитов Systemd

Systemd поддерживает несколько типов юнитов. Самые важные:

### `service`

Управление службами и демонами. Это самый частый тип.

### `target`

Группировка юнитов. Аналог старых runlevel, но более гибкий.

Примеры:

- `multi-user.target` — многопользовательский текстовый режим;
    
- `graphical.target` — графический режим;
    
- `network.target` — базовая сетевая готовность.
    

### `timer`

Запуск задач по расписанию. Часто рассматривается как замена `cron`.

### `mount`

Управление точками монтирования.

### `automount`

Автоматическое монтирование по обращению к каталогу.

### `path`

Запуск действия при изменении файла или каталога.

### `socket`

Активация сервиса через сокет. Сервис может запускаться только когда приходит соединение.

### Другие типы

Также существуют:

- `device`
    
- `scope`
    
- `slice`
    
- `swap`
    
- `snapshot`
    

Они используются реже, но важны для общей архитектуры Systemd.

## 4. Управление сервисами: типы и команды

### Типы сервисов (`Type=` в `[Service]`)

#### `simple`

Тип по умолчанию.  
Сервис считается запущенным сразу после старта процесса `ExecStart`.

Подходит для большинства обычных программ, которые работают на переднем плане.

#### `forking`

Используется для старых демонов, которые запускаются, а потом делают `fork` и уходят в фон.

#### `oneshot`

Подходит для задач, которые должны один раз выполниться и завершиться.

Например:

- создать директорию;
    
- смонтировать ресурс;
    
- настроить что-то при старте.
    

#### `idle`

Похож на `simple`, но запуск откладывается, пока не завершатся более приоритетные задачи.

### Ещё полезные типы

#### `notify`

Сервис сам сообщает Systemd, когда он действительно готов к работе. Это полезно для более точного управления состоянием службы.

#### `dbus`

Сервис считается готовым, когда он зарегистрировался на D-Bus.

## 5. Основные команды `systemctl`

### Базовое управление

Запустить сервис:

```bash
sudo systemctl start nginx
```

Остановить:

```bash
sudo systemctl stop nginx
```

Перезапустить:

```bash
sudo systemctl restart nginx
```

Посмотреть статус:

```bash
systemctl status nginx
```

### Автозагрузка

Включить в автозагрузку:

```bash
sudo systemctl enable nginx
```

Отключить:

```bash
sudo systemctl disable nginx
```

Включить и сразу запустить:

```bash
sudo systemctl enable --now nginx
```

Отключить и сразу остановить:

```bash
sudo systemctl disable --now nginx
```

### Применение изменений

После изменения файлов юнитов нужно перечитать конфигурацию:

```bash
sudo systemctl daemon-reload
```

Если юнит уже работает, после этого обычно ещё делают:

```bash
sudo systemctl restart myservice
```

### Просмотр состояния системы

Посмотреть все активные сервисы:

```bash
systemctl list-units --type=service
```

Посмотреть все юниты, включая неактивные:

```bash
systemctl list-unit-files
```

Проверить, включён ли сервис:

```bash
systemctl is-enabled nginx
```

Проверить, активен ли он:

```bash
systemctl is-active nginx
```

### Редактирование юнитов

Для безопасного изменения параметров пакета используют:

```bash
sudo systemctl edit nginx
```

Эта команда создаёт drop-in файл, обычно `override.conf`, и позволяет менять параметры без правки исходного пакетного файла.

## 6. Расположение файлов конфигурации

### Основные каталоги

1. **`/usr/lib/systemd/system`** — юниты, установленные пакетным менеджером.  
    В некоторых дистрибутивах может использоваться также `/lib/systemd/system`.
    
2. **`/etc/systemd/system`** — юниты, созданные или переопределённые администратором. Имеют приоритет над пакетными.
    
3. **`~/.config/systemd/user`** — пользовательские юниты.
    

### Приоритет

Если существует юнит с одинаковым именем и в `/usr/lib/systemd/system`, и в `/etc/systemd/system`, то приоритет имеет версия из `/etc`.

### Пользовательские юниты

Для работы с пользовательскими сервисами используют:

```bash
systemctl --user status
systemctl --user start myservice
systemctl --user enable myservice
```

Это удобно для процессов, не требующих прав root.

## 7. Дополнительные возможности Systemd

### Управление временем

Для времени используется `timedatectl`.

Посмотреть текущие настройки:

```bash
timedatectl
```

Список часовых поясов:

```bash
timedatectl list-timezones
```

Установить часовой пояс:

```bash
sudo timedatectl set-timezone Europe/Moscow
```

Включить синхронизацию времени:

```bash
sudo timedatectl set-ntp true
```

Systemd может использовать для этого сервис `systemd-timesyncd`.

---

### Управление сетью

Systemd может управлять сетью через `systemd-networkd`.

Конфигурации обычно лежат в:

```bash
/etc/systemd/network/
```

Файлы часто имеют расширение:

- `.network`
    
- `.netdev`
    
- `.link`
    

Пример очень простого `.network` файла:

```ini
[Match]
Name=enp0s3

[Network]
DHCP=yes
```

После изменения конфигурации может потребоваться перезапуск:

```bash
sudo systemctl restart systemd-networkd
```

### Уточнение

Во многих настольных системах вместо `systemd-networkd` может использоваться, например, `NetworkManager`.  
Поэтому наличие Systemd не означает, что именно он всегда управляет сетью.

## 8. Таймеры Systemd vs Cron

Таймеры считаются более гибкими и тесно интегрированными с Systemd.

### Преимущества таймеров

- связаны с конкретными юнитами;
    
- можно смотреть их статус через `systemctl`;
    
- единый механизм логирования;
    
- удобная зависимость от загрузки системы;
    
- можно запускать задачи и по календарю, и через промежутки времени.
    

### Виды времени

- `OnCalendar=` — запуск по календарю;
    
- `OnBootSec=` — через время после загрузки;
    
- `OnUnitActiveSec=` — через интервал после прошлого запуска;
    
- `OnStartupSec=` — через время после старта Systemd.
    

### Пример таймера

`/etc/systemd/system/backup.service`

```ini
[Unit]
Description=Run backup

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup.sh
```

`/etc/systemd/system/backup.timer`

```ini
[Unit]
Description=Backup every day

[Timer]
OnCalendar=daily
Persistent=true

[Install]
WantedBy=timers.target
```

Запуск:

```bash
sudo systemctl enable --now backup.timer
```

Посмотреть таймеры:

```bash
systemctl list-timers
```

### Проверка календарного выражения

```bash
systemd-analyze calendar "*:0/10"
```

Это помогает понять, как Systemd интерпретирует расписание.

### `Persistent=true`

Если система была выключена и пропустила момент запуска, после включения задача может быть выполнена. Это одно из удобных отличий от классического cron.

## 9. Монтирование и automount

### Mount-юниты

Они описывают точку монтирования и параметры монтирования.

Очень важное правило: имя mount-юнита должно строго соответствовать пути.

Пример:

- путь `/media/nfs`
    
- имя юнита `media-nfs.mount`
    

Проверить, как называется юнит для пути, можно мысленно заменить `/` на `-` и убрать начальный слэш.

### Пример mount-юнита

```ini
[Unit]
Description=Mount NFS share

[Mount]
What=server:/share
Where=/media/nfs
Type=nfs
Options=defaults

[Install]
WantedBy=multi-user.target
```

### Automount

Позволяет не держать файловую систему постоянно смонтированной.  
Она будет смонтирована при обращении к каталогу.

Это удобно:

- для сетевых ресурсов;
    
- для редко используемых путей;
    
- для ускорения загрузки системы.
    

Пример:

```ini
[Unit]
Description=Automount NFS share

[Automount]
Where=/media/nfs

[Install]
WantedBy=multi-user.target
```

### Преимущество automount

Если удалённый ресурс недоступен, система не обязательно будет тормозить загрузку так сильно, как при жёстком монтировании на старте.

## 10. Journald и логирование

Одно из ключевых отличий Systemd — встроенное логирование через **journald**.

### Просмотр логов

Показать весь журнал:

```bash
journalctl
```

Показать логи конкретного сервиса:

```bash
journalctl -u nginx
```

Показать только последние записи:

```bash
journalctl -u nginx -n 50
```

Следить в реальном времени:

```bash
journalctl -u nginx -f
```

Показать записи текущей загрузки:

```bash
journalctl -b
```

Показать ошибки:

```bash
journalctl -p err
```

### Преимущества journald

- единый формат;
    
- фильтрация по сервису, загрузке, приоритету, времени;
    
- интеграция с `systemctl`;
    
- удобно искать причины, почему сервис не поднялся.

## 11. Runlevel и target

В старых системах SysV использовались **runlevel**.  
В Systemd вместо них используются **target**.

Примерное соответствие:

- runlevel 3 → `multi-user.target`
    
- runlevel 5 → `graphical.target`
    
- runlevel 6 → `reboot.target`
    
- runlevel 1 → `rescue.target`
    

### Узнать текущий target

```bash
systemctl get-default
```

### Изменить target по умолчанию

Например, сделать текстовый режим:

```bash
sudo systemctl set-default multi-user.target
```

Вернуть графический:

```bash
sudo systemctl set-default graphical.target
```

## 12. Создание собственного сервиса

Пример простого собственного сервиса.

### Скрипт

`/usr/local/bin/hello.sh`

```bash
#!/usr/bin/env bash
while true; do
    echo "Hello from service"
    sleep 10
done
```

Сделать исполняемым:

```bash
chmod +x /usr/local/bin/hello.sh
```

### Юнит

`/etc/systemd/system/hello.service`

```ini
[Unit]
Description=Simple hello service

[Service]
Type=simple
ExecStart=/usr/local/bin/hello.sh
Restart=always

[Install]
WantedBy=multi-user.target
```

### Активация

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now hello.service
systemctl status hello.service
journalctl -u hello.service
```

## 13. Полезные параметры в `[Service]`

Часто используемые параметры:

- `ExecStart=` — основная команда запуска;
    
- `ExecStop=` — команда остановки;
    
- `ExecReload=` — команда перезагрузки конфигурации;
    
- `Restart=` — перезапуск при сбое;
    
- `RestartSec=` — пауза перед перезапуском;
    
- `User=` — от какого пользователя запускать;
    
- `Group=` — от какой группы;
    
- `WorkingDirectory=` — рабочая директория;
    
- `Environment=` — переменные окружения;
    
- `EnvironmentFile=` — переменные из файла;
    
- `RemainAfterExit=` — считать юнит активным даже после выхода процесса;
    
- `StandardOutput=` и `StandardError=` — настройка потоков вывода.
    

### Пример

```ini
[Service]
Type=simple
User=www-data
WorkingDirectory=/opt/app
Environment="PORT=8080"
ExecStart=/opt/app/server
Restart=on-failure
RestartSec=5
```

## 14. Полезные практические замечания

1. После изменения юнита почти всегда нужен:
    

```bash
sudo systemctl daemon-reload
```

2. Если сервис не стартует, сначала смотрят:
    

```bash
systemctl status имя.service
journalctl -u имя.service
```

3. Не стоит без необходимости редактировать пакетные юниты прямо в `/usr/lib/systemd/system`. Лучше использовать:
    

```bash
sudo systemctl edit имя.service
```

4. Для простых периодических задач timer часто удобнее, чем cron, если вся остальная инфраструктура уже завязана на Systemd.
    
5. Если сервис должен зависеть от сети, важно аккуратно выбирать между:
    

- `network.target`
    
- `network-online.target`
    

Потому что “сеть поднята” и “сеть полностью готова” — не всегда одно и то же.

## 15. Короткие выводы

1. **System V Init** — старый, простой и надёжный подход на bash-скриптах.
    
2. **Systemd** — современная система инициализации с юнитами, зависимостями и большим встроенным функционалом.
    
3. Основной инструмент управления — **`systemctl`**.
    
4. Самый распространённый тип юнита — **service**.
    
5. Для логов в Systemd используется **journald** и команда **`journalctl`**.
    
6. Таймеры Systemd могут заменять `cron` и тесно интегрированы с сервисами.
    
7. Юниты хранятся в `/usr/lib/systemd/system`, `/etc/systemd/system` и `~/.config/systemd/user`.
    
8. Для безопасной настройки лучше использовать override-файлы, а не править пакетные юниты напрямую.
    
9. Systemd даёт более удобное и централизованное управление системой, но ценой большей сложности.
